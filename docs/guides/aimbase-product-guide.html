1. Aimbase 개요

Aimbase는 애플리케이션에 AI 기능을 통합하기 위한 멀티테넌트 LLM 오케스트레이션 플랫폼입니다.

여러 LLM 프로바이더(Anthropic Claude, OpenAI GPT, Ollama 로컬 모델)를 통합 관리하며, RAG 지식 검색, MCP 기반 도구 호출, 워크플로우 자동화, 정책 엔진(가드레일)을 포함합니다. 클라이언트 애플리케이션은 REST API를 통해 이 기능들을 호출합니다.

주요 구성 요소

구성 요소설명
LLM 오케스트레이터멀티 프로바이더 LLM 라우팅, 세션 관리, 도구 호출 루프 관리
RAG 엔진문서 인제스션 → 청킹 → 임베딩 → 벡터 검색. 컨텍스트 보강 기반 답변 생성
MCP 허브외부 MCP 서버 연결 및 도구 라우팅. AI가 외부 시스템의 기능을 호출
워크플로우 엔진DAG 기반 멀티스텝 AI 파이프라인 실행
정책 엔진입력 필터링, PII 마스킹, 비용 제어, 도구 실행 승인 등 가드레일
Python 사이드카로컬 임베딩(BGE-M3), 고급 파싱, RAGAS 평가, 문서 생성 등 ML/NLP 처리
관리 콘솔 (FE)Connection, RAG, 정책, 워크플로우, 모니터링 등을 관리하는 웹 UI

Aimbase의 역할

클라이언트 앱은 비즈니스 로직에 집중하고, AI 관련 인프라(LLM 연동, 프롬프트 관리, RAG 파이프라인, 벡터DB, 안전 정책, 비용 추적, 감사 로깅)는 Aimbase가 담당합니다. 클라이언트 앱과 Aimbase 사이는 REST API로 통신하므로, LLM 프로바이더가 변경되거나 RAG 전략이 고도화되어도 클라이언트 앱의 코드 변경은 불필요합니다.

4. RAG 지식 검색 RAG A++

문서를 업로드하면 Aimbase가 파싱 → 청킹 → 임베딩 → 벡터DB 저장까지 처리합니다. 이후 사용자 질문 시 관련 문서 컨텍스트를 LLM에 자동 주입하여, 자사 데이터 기반의 정확한 답변을 생성합니다.

RAG 파이프라인

문서 업로드PDF, DOCX, TXT 등
→
고급 파싱구조 인식 분할
→
Contextual Chunking맥락 요약 첨부
→
임베딩BGE-M3 (로컬)
→
pgvector 저장HNSW 인덱스

RAG 기능 상세

기능설명
Contextual Retrieval 각 청크에 문서 전체 맥락 요약을 AI가 생성하여 함께 저장. 단편적 검색 결과 문제를 해결
Parent-Child 검색 작은 청크(Child)로 정밀 매칭 후, 해당 청크가 속한 큰 단위(Parent)를 반환. 정확도와 문맥 폭 동시 확보
Query Transform HyDE (가상 답변 생성 후 검색) + Multi-Query (질문을 여러 관점으로 재구성). 검색 재현율 향상
시맨틱 캐시 의미적으로 유사한 질문이 이전에 있었으면 캐시된 답변을 반환. LLM 호출 비용 절감 + 응답 속도 향상
Citation AI 답변에 출처 문서명 + 위치 정보를 마커로 표시. 근거 추적 가능
증분 인제스션 이미 처리된 문서는 건너뛰고 신규/변경분만 처리. 대량 문서 운영 시 효율적
RAGAS 평가 RAG 품질을 정량적으로 측정 (Faithfulness, Answer Relevancy, Context Precision). 테스트셋 기반 자동 평가
로컬 임베딩 BGE-M3 모델을 Python 사이드카에서 로컬 실행. 외부 API 호출 없이 다국어 임베딩. 1024차원 기본

5. MCP 도구 통합 MCP

MCP(Model Context Protocol)는 AI 모델이 외부 시스템의 기능(도구)을 호출할 수 있게 하는 표준 프로토콜입니다. 클라이언트 앱이 자신의 기능을 MCP 서버로 노출하면, Aimbase의 AI가 사용자 의도에 따라 해당 기능을 자동으로 호출합니다.

동작 방식

사용자 요청"주문 상태 알려줘"
→
Aimbase (LLM)의도 파악
도구 선택
→
앱 MCP 서버get_order_status
실행
→
Aimbase (LLM)결과 해석
자연어 응답 생성

MCP 관련 기능

기능설명
MCP 서버 관리외부 MCP 서버 URL을 등록/수정/삭제. SSE 프로토콜로 연결
도구 자동 검색등록된 MCP 서버에서 제공하는 도구 목록을 자동으로 가져와서 AI에 등록
도구 선택 제어요청별로 사용 가능한 도구를 필터링. 특정 도구만 허용하거나 차단 가능 (ToolFilterContext)
도구 호출 정책쓰기(W) 도구 실행 전 사용자 승인 요구, 특정 도구 차단 등 정책 엔진과 연동
내장 도구Aimbase 자체 제공 도구: 번역(TranslationTool), 알림(NotificationTool), 이미지 분석(ImageAnalysisTool) 등

MCP 도구 등록 예시

클라이언트 앱이 제공하는 기능을 MCP 도구로 정의합니다. AI는 이 정의를 보고 사용자 요청에 맞는 도구를 자동 선택합니다.

// MCP 도구 정의 예시 { "name": "search_products", "description": "상품을 검색합니다. 카테고리, 가격대, 키워드로 필터링 가능", "inputSchema": { "type": "object", "properties": { "keyword": { "type": "string", "description": "검색어" }, "category": { "type": "string", "description": "카테고리" }, "max_price": { "type": "integer", "description": "최대 가격" } }, "required": ["keyword"] } } // 사용자: "3만원 이하 블루투스 이어폰 찾아줘" // → LLM이 자동으로 search_products(keyword="블루투스 이어폰", max_price=30000) 호출

6. 워크플로우 엔진 Workflow

여러 AI 작업 단계를 DAG(방향 비순환 그래프)로 구성하여 순차 또는 병렬로 실행합니다. 각 단계(Step)는 LLM 호출, 도구 호출, 조건 분기 등을 수행할 수 있습니다.

워크플로우 구성 예시: 고객 문의 처리

Step 1문의 접수
→
Step 2의도 분류 (LLM)
→
Step 3지식 검색 (RAG)
→
Step 4답변 생성 (LLM)
→
Step 5티켓 생성 (MCP)

주요 기능

기능설명
비주얼 에디터React Flow 기반 드래그 앤 드롭 워크플로우 설계 (관리 콘솔 제공)
DAG 실행Kahn 알고리즘 위상 정렬. 의존 관계에 따라 순차/병렬 자동 결정
스텝 타입LLM 호출, 도구 호출, 조건 분기(router), 데이터 변환
스텝별 LLM 선택각 단계마다 다른 Connection(모델) 지정 가능. 분류는 Haiku, 생성은 Opus 등
변수 전달이전 스텝의 출력을 다음 스텝의 입력으로 자동 전달 (StepContext)
실행 이력모든 실행 기록 + 각 스텝별 입출력/소요시간/토큰 사용량 추적

7. 정책 엔진 (가드레일)

AI 요청의 입력·실행·출력 단계에서 안전과 비용을 통제하는 규칙 시스템입니다. 정책은 priority(우선순위) 내림차순으로 평가되며, 첫 번째 DENY 또는 REQUIRE_APPROVAL 결과에서 중단됩니다.

정책 유형

입력 정책

  • PII 마스킹 — 이름, 연락처, 주민번호 등 개인정보를 자동 마스킹 후 LLM에 전달
  • 금지어 차단 — 지정된 키워드/패턴 포함 시 요청 차단
  • 입력 길이 제한 — 최대 토큰 수 제한

실행 정책

  • 도구 승인 필요 — 쓰기(W) 도구 실행 전 사용자 확인 요구
  • 도구 호출 제한 — 최대 도구 호출 횟수 설정
  • 특정 도구 차단 — 지정된 도구의 사용을 금지

비용 정책

  • 일/월 비용 한도 — 설정 금액 도달 시 AI 호출 중단
  • 토큰 사용량 제한 — 요청당 또는 기간별 토큰 상한
  • 알림 임계값 — 비용 임계값 도달 시 관리자 알림

정책 평가 흐름

요청 수신 → 정책 #1 (priority 100) 평가 → ALLOW → 정책 #2 (priority 90) 평가 → ALLOW → 정책 #3 (priority 80) 평가 → DENY → 즉시 중단, 요청 차단 // 판정 결과: ALLOW(허용), DENY(차단), REQUIRE_APPROVAL(승인 대기)

9. API 연동 방식

클라이언트 앱은 다음 3가지 방식으로 Aimbase와 연동할 수 있습니다.

방식 1: 오케스트레이션 API (기본)

가장 일반적인 방식. 사용자 메시지를 보내면 LLM 응답(+ 도구 호출 결과 포함)을 받습니다.

POST /api/v1/orchestrate { "session_id": "session-uuid", "connection_id": "conn-uuid", "message": "사용자 입력 메시지", "role_id": "role-uuid" // 선택: AI 역할 지정 } // 응답: AI 답변 텍스트 + 도구 호출 결과 + 토큰 사용량

방식 2: RAG 검색 API

RAG 검색만 독립적으로 호출. 문서 기반 질의응답이 필요할 때 사용합니다.

POST /api/v1/knowledge-sources/{id}/search { "query": "검색 질문", "top_k": 5 } // 응답: 관련 문서 청크 목록 (유사도 점수 포함)

방식 3: 워크플로우 실행 API

등록된 워크플로우를 트리거합니다. 멀티스텝 자동화에 사용합니다.

POST /api/v1/workflows/{id}/run { "input": { "text": "처리할 데이터" } } // 응답: 워크플로우 실행 결과 (각 스텝별 출력 포함)
인증 모든 API는 JWT 토큰 또는 API Key 인증이 필요합니다. 요청 헤더에 Authorization: Bearer {token} 또는 X-API-Key: {key}를 포함합니다.

11. 멀티테넌시

Aimbase는 Database-per-Tenant 아키텍처를 사용합니다. 각 테넌트(고객사)는 독립된 데이터베이스를 가지며, 데이터가 물리적으로 분리됩니다.

항목설명
테넌트 격리테넌트별 독립 DB. A사의 문서/대화/정책이 B사에 노출되지 않음
테넌트별 설정LLM Connection, 정책, RAG 설정, MCP 도구를 테넌트 단위로 독립 구성
비용 분리 추적테넌트별 토큰 사용량, API 호출 수, 비용을 분리 집계
플랫폼 관리Super Admin이 /api/v1/platform/** 엔드포인트로 전체 테넌트를 관리 (Master DB)
마이그레이션Flyway 기반. Master DB와 Tenant DB 마이그레이션 경로 분리
