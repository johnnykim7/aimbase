## 작성 원칙

1. **상황→행동→기대결과** 형식으로 테스트 케이스 정의
2. FSM 상태 전환, 이벤트 발행, 정책 평가를 필수 포함
3. 정상/에러/경계값 케이스를 균형 있게 포함
4. TC-ID 형식: `TC-[모듈약어]-[순번]`
5. 관련 ID(PRD, BIZ, POL)를 명시하여 추적성 확보

## 작성 가이드

### TC-ID 규칙
- `TC-[모듈약어]-[순번]`
- 모듈약어: ORC(오케스트레이터), SES(세션), LLM, TOOL, TREG(도구레지스트리), POL(정책), RAG, WF(워크플로우), TNT(테넌트), FE(프론트엔드), **PY-RAG, PY-EVAL, PY-SAFE(Python 사이드카)**

### 관련 ID 매핑
- `PRD-xxx`: 기능 요구사항 - Spring (T1-1)
- `PY-xxx`: 기능 요구사항 - Python (T1-1)
- `BIZ-xxx`: 비즈니스 규칙 (T1-3)
- `POL-xxx`: 정책 (T1-4)

### 필수 포함 패턴
- ✅ 정상 케이스
- ✅ 에러/예외 케이스
- ✅ 경계값 (최대 5회, TTL 만료 등)
- ✅ FSM 상태 전환 (WorkflowRun, KnowledgeSource, Tenant)
- ✅ 정책 규칙 평가 (각 PolicyRuleType별)
- ✅ 멀티테넌트 격리
