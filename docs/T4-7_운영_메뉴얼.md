## 2. 일상 운영

### 서버 기동/중지/재시작

```bash
# 전체 기동
cd backend && docker compose up -d

# 전체 중지
docker compose down

# BE 재시작만
docker compose restart api

# BE 로그 확인
docker compose logs -f api
```

### Health Check

| 대상 | 엔드포인트 | 정상 응답 | 확인 주기 |
|------|----------|---------|----------|
| API Server | GET /actuator/health | `{"status":"UP"}` | 1분 |
| PostgreSQL | 포함 (health.db) | status=UP | 1분 |
| Redis | GET /actuator/health | components.redis=UP | 1분 |
| Prometheus | GET /actuator/prometheus | 메트릭 텍스트 | 5분 |

### 로그 위치

| 로그 | 위치 | 용도 |
|------|------|------|

```bash

# 정책 평가 로그
```

## 4. 장애 대응

### 장애 등급

| 등급 | 설명 | 대응 시간 | 예시 |
|------|------|----------|------|
| P1 | 전체 서비스 불가 | 즉시 | API 서버 다운, Master DB 장애 |
| P2 | 핵심 기능 장애 | 30분 내 | 채팅 불가, 워크플로우 실패 |
| P3 | 부가 기능 장애 | 4시간 내 | 모니터링 오류, 특정 페이지 오류 |
| P4 | 경미한 이슈 | 다음 배포 | UI 깨짐, 로그 누락 |

### 장애 대응 절차

```
1. 장애 감지 (모니터링/사용자 리포트)
   │
   ▼
2. 장애 등급 판단 (P1~P4)
   │
   ▼
3. 원인 분석
   ├── 로그 확인 (docker compose logs)
   ├── Health Check (GET /actuator/health)
   └── DB 상태 확인 (psql)
   │
   ▼
4. 조치
   ├── 서비스 재시작 (docker compose restart)
   ├── 롤백 (이전 버전 JAR)
   └── DB 복구 (백업 복원)
   │
   ▼
5. 확인 (스모크 테스트 T4-4 실행)
   │
   ▼
6. 사후 분석 (RCA 작성)
```

### 롤백 절차

```bash
# 1. 현재 상태 백업
docker compose exec postgres-master pg_dump -U platform llm_platform_master > backup_$(date +%Y%m%d).sql

# 2. 이전 버전으로 교체
docker compose down api
# 이전 JAR로 교체
docker compose up -d api

# 3. 스모크 테스트 실행
curl http://localhost:8080/actuator/health
```

## 5. 주요 장애 시나리오별 대응

### API 서버 무응답
1. `docker compose ps` → api 상태 확인
2. `docker compose logs --tail=100 api` → 에러 로그 확인
3. `docker compose restart api` → 재시작
4. 지속 시 → JVM 메모리 확인, 힙 덤프 분석

### DB 접속 실패
1. `docker compose ps` → postgres 상태 확인
2. `docker exec postgres-master pg_isready` → DB 상태
3. 연결 풀 소진 시 → 애플리케이션 재시작
4. 디스크 부족 시 → 볼륨 정리

### Redis 장애
1. `docker exec redis redis-cli ping` → 연결 확인
2. 세션 데이터 소실 → 사용자 재로그인 필요
3. Rate Limiter 초기화됨 → 일시적 제한 해제 주의
4. `docker compose restart redis` → 재시작

### LLM API 오류
1. 외부 서비스(OpenAI/Anthropic) 상태 확인
2. API 키 유효성/쿼터 확인
3. 대안 모델로 라우팅 변경 (routing_config)
4. Ollama 로컬 모델로 fallback

## 7. 백업/복구

### DB 백업

```bash
# Master DB 백업
docker exec postgres-master pg_dump -U platform llm_platform_master > master_backup_$(date +%Y%m%d).sql

# Tenant DB 백업 (특정 테넌트)
docker exec postgres-master pg_dump -U platform llm_platform_tenant_dev > tenant_dev_backup_$(date +%Y%m%d).sql
```

### DB 복구

```bash
# Master DB 복구
docker exec -i postgres-master psql -U platform llm_platform_master < master_backup_20260310.sql

# Tenant DB 복구
docker exec -i postgres-master psql -U platform llm_platform_tenant_dev < tenant_dev_backup_20260310.sql
```
