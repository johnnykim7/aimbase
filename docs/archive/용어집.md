# Aimbase 주요 용어집

> SK브로드밴드 설명회 참고용 — 멀티에이전트 오케스트레이션 플랫폼 핵심 용어 정리

---

## 1. 오케스트레이션 핵심

| 용어 | 설명 |
|------|------|
| **DAG (Directed Acyclic Graph)** | 방향성 비순환 그래프. 워크플로우의 스텝과 의존관계를 표현하는 자료구조. 순환(루프)이 없어 실행 순서가 명확하게 결정됨 |
| **위상 정렬 (Topological Sort)** | DAG에서 스텝 실행 순서를 결정하는 알고리즘. Aimbase는 Kahn's Algorithm을 사용하여 의존성이 충족된 스텝부터 순서대로 실행 |
| **Kahn's Algorithm** | 진입 차수(in-degree)가 0인 노드부터 큐에 넣고, 실행 후 후속 노드의 진입 차수를 감소시키는 방식의 위상 정렬 알고리즘. O(V+E) 시간 복잡도 |
| **오케스트레이션 (Orchestration)** | 여러 LLM, 도구, 정책, RAG 등을 하나의 요청 흐름으로 조율하는 것. 세션→정책평가→RAG주입→LLM호출→도구루프→가드레일→응답의 파이프라인을 관리 |
| **워크플로우 (Workflow)** | 복수의 스텝을 DAG로 정의한 자동화 프로세스. LLM 호출, 도구 실행, 조건 분기, 병렬 처리, 승인 게이트 등을 조합 |

## 2. LLM & 어댑터

| 용어 | 설명 |
|------|------|
| **LLM (Large Language Model)** | 대규모 언어 모델. Anthropic Claude, OpenAI GPT, Ollama(로컬) 등 |
| **어댑터 패턴 (Adapter Pattern)** | 프로바이더(Anthropic/OpenAI/Ollama)마다 다른 API를 통일된 인터페이스(`LLMAdapter`)로 추상화하는 설계 패턴. 새 프로바이더 추가 시 어댑터만 구현하면 됨 |
| **모델 라우팅 (Model Routing)** | 요청을 어떤 LLM 프로바이더/모델로 보낼지 결정하는 전략. round-robin(균등 분배), cost-optimized(비용 최적), latency(지연 최소) 3가지 전략 지원 |
| **Connection** | LLM 프로바이더와의 연결 설정 (API 키, 모델명, 엔드포인트 등). DB에 저장되며 런타임에 동적 추가/변경 가능 |
| **Tool Use (도구 호출)** | LLM이 외부 도구 실행을 요청하는 메커니즘. LLM 응답에 `tool_use` 블록이 포함되면 해당 도구를 실행하고 결과를 다시 LLM에 전달 |
| **Tool Use Loop (도구 호출 루프)** | LLM→도구호출→결과→LLM 재호출을 반복하는 과정. 무한 루프 방지를 위해 최대 5회 제한 (BIZ-001) |
| **Tool Choice (도구 선택 제어)** | LLM의 도구 사용을 제어하는 파라미터. `auto`(자율), `none`(미사용), `required`(강제), `{toolName}`(특정 도구 강제) |
| **FinishReason** | LLM 응답 종료 사유. `END`(정상 완료), `TOOL_USE`(도구 호출 필요), `MAX_TOKENS`(토큰 초과), `ERROR`(오류) |
| **Structured Output (구조화된 출력)** | LLM 응답을 JSON Schema에 맞춰 구조화된 데이터로 반환받는 기능. 폼 추출, 데이터 파싱 등에 활용 |

## 3. RAG (검색 증강 생성)

| 용어 | 설명 |
|------|------|
| **RAG (Retrieval-Augmented Generation)** | 외부 문서에서 관련 정보를 검색하여 LLM 프롬프트에 주입함으로써 응답 정확도를 높이는 기법. 환각(hallucination) 감소에 효과적 |
| **임베딩 (Embedding)** | 텍스트를 고차원 벡터(숫자 배열)로 변환하는 것. 의미적 유사도 계산의 기반. Aimbase는 KoSimCSE(768차원) 또는 OpenAI text-embedding-3-small(1536차원) 사용 |
| **시맨틱 청킹 (Semantic Chunking)** | 문서를 고정 크기가 아닌 의미 단위로 분할하는 기법. 임베딩 유사도 기반으로 문맥이 전환되는 지점을 감지하여 분할 |
| **하이브리드 검색 (Hybrid Search)** | BM25(키워드 매칭) + 벡터 검색(의미 유사도)을 결합하여 검색 품질을 높이는 방식 |
| **BM25** | 키워드 빈도 기반의 전통적 검색 알고리즘. 정확한 키워드 매칭에 강점 |
| **코사인 유사도 (Cosine Similarity)** | 두 벡터 간 각도를 측정하여 의미적 유사도를 계산하는 방법. 1에 가까울수록 유사 |
| **RRF (Reciprocal Rank Fusion)** | 여러 검색 결과의 순위를 통합하는 알고리즘. `score = 1/(k + rank)`로 계산하여 키워드 검색과 벡터 검색 결과를 결합 |
| **HNSW (Hierarchical Navigable Small World)** | pgvector의 근사 최근접 이웃 탐색 인덱스. 대량 벡터에서 빠른 유사도 검색을 가능하게 하는 그래프 기반 인덱스 |
| **pgvector** | PostgreSQL의 벡터 검색 확장. 별도의 벡터 DB 없이 기존 PostgreSQL에서 벡터 연산 수행. 운영 복잡도 최소화 |
| **리랭킹 (Reranking)** | 초기 검색 결과를 Cross-Encoder 모델로 재평가하여 정확도를 높이는 후처리 단계 |
| **Cross-Encoder** | 쿼리-문서 쌍을 함께 입력받아 관련성 점수를 직접 계산하는 모델. Bi-Encoder보다 정확하지만 느림. 리랭킹에 사용 |
| **HyDE (Hypothetical Document Embedding)** | 쿼리에 대한 가상 답변을 LLM으로 생성한 뒤, 그 답변을 임베딩하여 검색하는 기법. 질문과 문서의 의미 갭을 줄임 |
| **Multi-Query** | 원본 질문을 3~5가지 다른 표현으로 변환하여 병렬 검색 후 결과를 통합하는 기법 |

## 4. MCP (Model Context Protocol)

| 용어 | 설명 |
|------|------|
| **MCP (Model Context Protocol)** | Anthropic이 주도하는 LLM 도구 통합 표준 프로토콜. 도구 정의, 탐색, 실행을 표준화하여 다양한 도구를 LLM에 연결 |
| **MCP Server** | MCP 프로토콜을 구현한 도구 서버. Aimbase는 RAG, Safety, Evaluation, Agent 4개의 Python MCP 서버를 운영 |
| **FastMCP** | Python용 MCP 서버 프레임워크. `@mcp.tool()` 데코레이터로 간편하게 도구 등록 |
| **SSE (Server-Sent Events)** | 서버→클라이언트 단방향 실시간 스트리밍 프로토콜. LLM 응답 스트리밍과 MCP 서버 통신에 사용 |
| **Tool Registry (도구 레지스트리)** | 빌트인 도구와 MCP 도구를 통합 관리하는 중앙 저장소. 도구 자동 탐색(discovery) 지원 |

## 5. 정책 엔진 & 거버넌스

| 용어 | 설명 |
|------|------|
| **정책 엔진 (Policy Engine)** | LLM 호출/액션 실행 전에 규칙을 평가하여 허용/거부/변환을 결정하는 엔진 |
| **DENY** | 요청을 즉시 거부하는 정책 규칙. 평가 즉시 중단 |
| **REQUIRE_APPROVAL** | 관리자 승인이 필요한 정책 규칙. 승인 대기 상태(PendingApproval) 생성 |
| **RATE_LIMIT** | 요청 빈도를 제한하는 정책 규칙. 슬라이딩 윈도우 알고리즘으로 구현 |
| **TRANSFORM** | 데이터를 변환하는 정책 규칙. 주로 PII(개인정보) 마스킹에 사용 |
| **슬라이딩 윈도우 (Sliding Window)** | 시간 창(예: 60초) 내 요청 횟수를 추적하여 빈도를 제한하는 알고리즘. 고정 윈도우 대비 경계값 문제 없음 |
| **PII (Personally Identifiable Information)** | 개인식별정보. 주민등록번호, 전화번호, 이메일, 계좌번호, 주소 등 |
| **Presidio** | Microsoft의 오픈소스 PII 탐지/마스킹 엔진. Aimbase는 한국어 커스텀 인식기(주민번호, 전화번호 등)를 추가하여 사용 |
| **가드레일 (Guardrails)** | LLM 출력의 안전성을 검증하는 장치. 주제 이탈, 유해 콘텐츠, PII 노출 등을 탐지 |
| **RBAC (Role-Based Access Control)** | 역할 기반 접근 제어. 8개 리소스 x 2개 권한(read/write) = 16개 권한 조합으로 세밀한 접근 통제 |
| **감사 로그 (Audit Log)** | 모든 주요 이벤트(LLM 호출, 도구 사용, 정책 판정, 사용자 액션)를 기록하는 추적 로그. 규정 준수 및 사고 분석에 활용 |

## 6. 멀티테넌시

| 용어 | 설명 |
|------|------|
| **멀티테넌시 (Multi-Tenancy)** | 하나의 시스템으로 여러 고객(테넌트)을 독립적으로 서비스하는 아키텍처 |
| **Database-per-Tenant** | 테넌트마다 독립된 PostgreSQL 데이터베이스를 할당하는 격리 전략. 물리적 데이터 분리로 최고 수준의 보안 보장 |
| **TenantContext** | 현재 요청의 테넌트 ID를 ThreadLocal에 바인딩하여 전파하는 메커니즘. Virtual Thread에서도 안전하게 동작 |
| **Dynamic DataSource Routing** | 요청의 테넌트 ID에 따라 해당 테넌트의 DB로 자동 라우팅하는 기능. 재시작 없이 새 테넌트 즉시 추가 가능 |
| **Master DB** | 테넌트 메타데이터, 구독 정보, 전역 설정을 저장하는 중앙 데이터베이스 |
| **쿼터 (Quota)** | 테넌트별 리소스 사용 한도. 월간 토큰 쿼터, 최대 연결 수, 지식소스 수, 워크플로우 수 등 |
| **Flyway** | DB 스키마 마이그레이션 도구. Master DB와 Tenant DB를 별도 경로로 관리 |

## 7. 동시성 & 성능 최적화

| 용어 | 설명 |
|------|------|
| **Virtual Threads (가상 스레드)** | Java 21의 경량 스레드. OS 스레드 1개로 수천 개의 가상 스레드를 구동. LLM 호출 같은 I/O 바운드 작업에 최적 |
| **CompletableFuture** | Java의 비동기 연산 결과를 나타내는 객체. LLM 호출, 액션 실행 등의 비동기 처리에 사용 |
| **병렬 스텝 (PARALLEL Step)** | 워크플로우에서 의존관계 없는 여러 스텝을 동시에 실행하는 스텝 타입 |
| **컨텍스트 윈도우 트리밍 (Context Window Trimming)** | 대화 이력이 토큰 한도(기본 100K)를 초과할 때 오래된 메시지를 자동 제거하는 기법. 시스템 메시지는 항상 보존 |
| **SSE 스트리밍** | LLM 응답을 토큰 단위로 실시간 전송. 전체 응답 대기 없이 즉시 출력 시작 → 사용자 체감 지연 최소화 |
| **어댑터 캐시** | LLM 클라이언트 인스턴스를 Connection별로 캐싱. 매 요청마다 클라이언트 재생성 방지 |

## 8. 평가 & 품질 관리

| 용어 | 설명 |
|------|------|
| **RAGAS** | RAG 시스템 품질 평가 프레임워크. Faithfulness(충실도), Answer Relevancy(답변 관련성), Context Precision(컨텍스트 정확도), Context Recall(컨텍스트 재현율) 4가지 메트릭 |
| **Faithfulness (충실도)** | LLM 응답이 검색된 컨텍스트에 얼마나 충실한가. 1.0에 가까울수록 환각 없음 |
| **DeepEval** | LLM 출력 품질 평가 프레임워크. 환각(Hallucination), 유해성(Toxicity), 편향성(Bias) 점수를 0.0~1.0으로 측정 |
| **A/B 테스트 (Prompt Comparison)** | 두 프롬프트의 LLM 응답 품질을 비교하여 더 나은 프롬프트를 선택하는 회귀 테스트 |

## 9. 에이전트 추론 패턴

| 용어 | 설명 |
|------|------|
| **LangGraph** | 상태 기반 에이전트 추론 프레임워크. 노드(추론 단계)와 엣지(전환 조건)로 복잡한 추론 체인을 구성 |
| **ReAct (Reasoning + Acting)** | 추론(Thought)→행동(Action)→관찰(Observation)을 반복하는 에이전트 패턴. 도구 활용과 추론을 교차 수행 |
| **Plan and Execute** | 전체 계획을 먼저 수립한 뒤 단계적으로 실행하는 에이전트 패턴. 복잡한 작업에 적합 |
| **Reflection** | LLM이 자신의 출력을 검토하고 자가 수정하는 패턴. 응답 품질 향상에 활용 |

## 10. 인프라 & 모니터링

| 용어 | 설명 |
|------|------|
| **Python 사이드카 (Sidecar)** | 메인 Java 백엔드와 함께 실행되는 Python 마이크로서비스. AI 특화 라이브러리(RAGAS, Presidio, LangGraph 등) 활용을 위해 분리 |
| **Prometheus** | 시계열 메트릭 수집/저장 시스템. LLM 호출 수, 지연시간, 토큰 사용량, 정책 위반 등을 수집 |
| **Micrometer** | Java 메트릭 수집 라이브러리. Prometheus 등 다양한 백엔드로 메트릭 전송 |
| **HikariCP** | 고성능 JDBC 커넥션 풀. Virtual Threads와 연동하여 DB 연결 효율 극대화 |
| **Spring Boot Actuator** | 애플리케이션 상태/메트릭/헬스체크를 노출하는 Spring 모듈. `/actuator/prometheus` 엔드포인트 제공 |

## 11. 워크플로우 스텝 유형

| 스텝 타입 | 설명 | 활용 예시 |
|----------|------|----------|
| **LLM_CALL** | 언어 모델 호출 | 텍스트 생성, 분류, 요약 |
| **TOOL_CALL** | 외부 도구 실행 | DB 조회, API 호출, 파일 처리 |
| **CONDITION** | 조건 분기 | 이전 스텝 결과에 따라 다른 경로 실행 |
| **PARALLEL** | 병렬 실행 | 여러 LLM을 동시 호출하여 결과 비교 |
| **HUMAN_INPUT** | 승인 게이트 | 고위험 작업 전 관리자 승인 대기 |
| **ACTION** | 외부 시스템 실행 | DB 쓰기, Slack 알림, WebSocket 발송 |

## 12. 상태 머신 (FSM) 주요 상태

| 엔티티 | 상태 전이 |
|--------|----------|
| **Connection** | `disconnected` → `connected` → `error` |
| **KnowledgeSource** | `idle` → `syncing` → `completed` / `error` |
| **WorkflowRun** | `running` → `pending_approval` / `completed` / `failed` |
| **Tenant** | `provisioning` → `active` → `suspended` → `deleted` |
