# 원본 요구사항 — 성능/퀄리티 메커니즘 + 프로바이더 확장

> **출처**: openclaude(Claude Code CLI 오픈소스) vs Aimbase 전수 비교 — 2026-04-07 갭 분석 결과
> **작성일**: 2026-04-07
> **CR**: CR-031 (성능/퀄리티), CR-032 (프로바이더 확장)

---

## 배경

openclaude 소스코드와 Aimbase를 전수 비교한 결과, LLM 출력 품질과 운영 비용에 직접 영향을 미치는 내부 메커니즘 6개와 프로바이더 확장 6개가 미구현 상태로 식별되었다.

- 프로바이더를 아무리 늘려도 내부 품질 메커니즘이 없으면 장시간 대화 품질 급락, 불필요한 비용 발생
- Aimbase는 Anthropic/OpenAI/Ollama 3개만 지원. 엔터프라이즈에서 Bedrock/Vertex/Gemini 필수

## CR-031: 성능/퀄리티 메커니즘 (6개)

### 1. Post-Compact Recovery (압축 후 컨텍스트 복구) ★★★★★

**현재**: POST_COMPACT 훅은 알림만. 손실된 컨텍스트 복구 메커니즘 없음.

**openclaude 구현**:
- POST_COMPACT_MAX_FILES_TO_RESTORE = 5 (각 5K tokens)
- POST_COMPACT_TOKEN_BUDGET = 50_000 (파일 복구 총 예산)
- POST_COMPACT_MAX_TOKENS_PER_SKILL = 5_000 (스킬당)
- POST_COMPACT_SKILLS_TOKEN_BUDGET = 25_000 (스킬 복구 총 예산)
- microcompact state reset
- memory file cache 초기화 → MEMORY.md 재로드
- system prompt sections 정리
- 최근 참조 파일 5개 + 스킬 5개 자동 재주입

**Aimbase 구현 방향**:
- ContextWindowManager.trimWithState() 이후 PostCompactRecoveryService 호출
- ToolExecutionLog에서 최근 참조 파일 top-5 추출 → FileReadTool로 재주입
- LONG_TERM 메모리 재확인 → system message 갱신
- MCP 도구 스키마 재주입

**영향**: 장시간 대화에서 압축 후 "같은 질문 반복" / "엉뚱한 답변" 방지.

### 2. MICRO_COMPACT 캐시 최적화 ★★★(비용)

**현재**: ConversationSummarizer.microSummarize()가 매번 Haiku 호출 (~$0.0015/회).

**openclaude 구현**:
- 캐시 기반 0비용: cache_edits API로 메시지 content 변경 없이 캐시 레벨에서 tool result 삭제
- time-based trigger: 마지막 응답 후 N분 경과 시 cold cache 대상만 직접 clear
- 마커 대체: `[Old tool result content cleared]`

**Aimbase 구현 방향**:
- 방안 A: LLM 호출 없이 직접 축약 — 오래된 tool result → 마커 대체, 최근 N개만 보존
- 방안 B: 요약 캐싱 — 동일 content hash의 요약을 ResponseCacheService에 저장

### 3. Extract Memories 자동화 ★★★★★

**현재**: 수동 API (`POST /api/v1/memories`)로만 저장. 자동 추출 전무.

**openclaude 구현**:
- query loop 완료 후 forked agent가 자동 분석
- 4가지 타입: task-specific, person context, pattern/best practice, personal preferences
- 기존 메모리 중복 검사 → 업데이트 또는 신규 생성
- fire-and-forget async (transcript 미기록)
- 5 turns max, turn 기반 throttle

**Aimbase 구현 방향**:
- OrchestratorEngine.chat() 응답 완료 후 (SESSION_END 훅):
  1. 대화 길이 >= 5 turn이면 트리거
  2. 비동기(Virtual Thread)로 Haiku 호출
  3. 프롬프트: "이 대화에서 향후 재사용할 핵심 정보 추출"
  4. 결과를 LONG_TERM 메모리로 자동 저장 (ConversationMemoryEntity)
  5. 중복 체크: 기존 LONG_TERM 메모리와 cosine similarity 비교
  6. 다음 SESSION_MEMORY 압축(Level 3)에서 활용

### 4. Adaptive Thinking ★★★★

**현재**: `extendedThinking: true/false` 이진 선택만.

**openclaude 구현**:
- type: 'adaptive' (모델이 자동 budget 결정)
- type: 'enabled' + budgetTokens (고정 budget)
- type: 'disabled'
- Claude 4.6+ 모델에서만 adaptive 지원

**Aimbase 구현 방향**:
- ModelConfig에 ThinkingMode enum 추가 (DISABLED, ENABLED, ADAPTIVE)
- AnthropicAdapter에서 thinking 파라미터 분기

### 5. Tool Result 지능형 축약 ★★★

**현재**: 80KB 초과 시 단순 truncate. 의미 있는 내용이 잘릴 수 있음.

**openclaude 구현**: COMPACTABLE_TOOLS별 선별적 축약 (Read, Bash, Grep, Glob, WebSearch 등)

**Aimbase 구현 방향**:
- ToolCallHandler 결과 저장 시 도구별 축약 전략 적용
- 파일 읽기: 헤더 + 핵심 섹션만 보존
- Grep: 매칭 라인 + 컨텍스트만, 중복 제거
- Bash: stdout 마지막 50줄 + exit code
- full_output + compact_output 이중 저장

### 6. Agent 진행 요약 (30초 주기) ★★

**현재**: SubagentRunner는 최종 결과만 반환. 중간 진행 상황 불투명.

**openclaude 구현**: 30초마다 forked agent로 3-5단어 현재형 요약

**Aimbase 구현 방향**:
- SubagentRunner.run() 내부 ScheduledExecutorService 30초 주기
- 최근 도구 호출 기록 기반 요약 (Haiku 1회)
- SubagentRunEntity.progressSummary 필드 추가
- SSE/WebSocket으로 FE에 푸시

## CR-032: 프로바이더 확장

### A. 프로바이더 확장 (6개)

| # | 프로바이더 | 우선순위 |
|---|----------|---------|
| 1 | OpenAI 호환 범용 shim (DeepSeek, Groq, Mistral, LM Studio 등) | 🔴 최우선 |
| 2 | AWS Bedrock | 🔴 높음 |
| 3 | Google Vertex AI | 🔴 높음 |
| 4 | Gemini (shim 기반) | 🔴 높음 |
| 5 | Azure Foundry | 🟡 중간 |
| 6 | GitHub Models | 🟡 중간 |

### B. 에이전트별 프로바이더 라우팅

**현재**: ModelRouter는 요청 복잡도 기반 (SIMPLE→haiku, COMPLEX→opus).
에이전트 타입별로 다른 모델/프로바이더 쓰는 기능 없음.

**구현 방향**:
- SubagentRequest에 preferredConnectionId / preferredModel 필드 추가
- AgentCallStepExecutor에서 스텝별 connection_id 지정 가능
- Workflow AGENT_CALL 노드 설정에 라우팅 규칙 추가
- FE ConfigPanel에 에이전트별 모델 선택 드롭다운
