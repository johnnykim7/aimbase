# 1. 문서 성격과 사용 목적

본 문서는 두 가지 목적을 동시에 가진다.

첫째, **기획서**로서 이번 변경의 목적, 범위, 원칙, 완료 기준을 명확히 한다.

둘째, **선행 개발 설계서**로서 개발팀이 세부 구현에 들어가기 전에 반드시 공유해야 할 아키텍처 방향, 계층 분리 원칙, 구현 순서, 위험 통제 포인트를 정의한다.

즉, 본 문서는 화면 요구사항이나 API 목록만 정리한 문서가 아니라, 다음 질문에 대한 공통 답변을 제공하는 문서다.

- 왜 이 작업을 하는가
- 정확히 무엇을 만들고 무엇을 만들지 않는가
- Claude, Claude CLI, ClaudeTool, Aimbase의 관계는 무엇인가
- 1단계에서 어디까지 내재화하고 어디까지 런타임에 남길 것인가
- 어떤 순서로 구현해야 재작업과 오해를 줄일 수 있는가

# 2. 핵심 배경

## 2.1 출발점

Aimbase가 ClaudeTool을 사용하게 된 최초의 이유는 비용 구조였다.

ClaudeTool은 Aimbase 내부에서 Claude CLI를 호출하는 통합 모듈이며, Claude CLI는 정액제 기반 사용이라는 현실적인 장점이 있었다. 초기에는 이 비용 구조 때문에 채택되었지만, 실제 사용 과정에서 Claude CLI가 단순 모델 호출기가 아니라 다음과 같은 실행 레이어라는 점이 중요하게 드러났다.

- 대화형 작업 흐름
- 도구 사용의 자연스러운 연결
- 세션 지속과 이어서 작업하는 감각
- 워크스페이스를 다루는 방식
- 문맥을 쌓고 정리하는 운영 방식
- 사람처럼 보이는 작업 진행 UX

즉, 출발은 비용이었지만 실제 인사이트는 **Claude CLI가 Claude를 감싸는 실행/운영 레이어**에 있었다.

## 2.2 OpenClaude를 본 이유

OpenClaude를 검토한 목적은 Claude 자체를 복제하기 위해서가 아니다.

또한 몇몇 개별 도구를 그대로 포팅하기 위해서만도 아니다.

핵심 목적은 **Claude CLI가 왜 그렇게 자연스럽고 만능적으로 느껴지는지, 그 외곽 레이어의 구조적 힌트를 확인하는 것**이었다.

다만 분석 결과, OpenClaude는 일부 도구 입력 스키마나 CLI형 UX 감각을 참고할 수는 있으나, 이번 프로젝트의 본체가 되는 Session Meta, Context Assembly, Runtime 재배치, Domain 연결 구조까지 제공하는 핵심 설계 출처는 아니었다.

따라서 이번 프로젝트의 성격은 다음과 같이 정리한다.

- OpenClaude 포팅 프로젝트가 아니다.
- Claude 자체 복제 프로젝트가 아니다.
- **ClaudeTool 내부에 묻려 있는 운영 인사이트의 일부를 Aimbase 상위 레이어로 끌어올리는 프로젝트**다.

# 3. 용어 정의

이 문서에서는 아래 용어를 고정해서 사용한다.

## 3.1 Claude
Claude는 LLM 모델 자체를 의미한다.

## 3.2 Claude CLI
Claude CLI는 Claude를 감싸는 실행 레이어를 의미한다. 여기에는 대화형 UX, 도구 호출 흐름, 세션 지속, 워크스페이스 접근, 문맥 운영 방식이 포함된다.

## 3.3 ClaudeTool
ClaudeTool은 Aimbase 내부에서 Claude CLI를 호출하는 통합 모듈이다.

즉,

**ClaudeTool = Aimbase 내부의 Claude CLI 런타임 연결 지점**

이다.

## 3.4 OpenClaude
OpenClaude는 Claude CLI와 유사한 구조를 가진 오픈소스 레퍼런스다. 일부 도구 스키마나 CLI 흐름을 참고할 수 있으나, 이번 변경의 핵심 출처는 아니다.

## 3.5 Aimbase
Aimbase는 범용 AI 제품이 아니다.

Aimbase는 AXOPM, 입찰 시스템, WMS, OMS, Mall 등 **도메인 솔루션에 AI를 반복 가능하게 이식하기 위한 상위 운영 레이어**다.

# 4. 이번 변경의 본질

이번 변경의 본질은 ClaudeTool을 제거하는 것이 아니다.

또한 Claude 자체의 지능을 복제하는 것도 아니다.

이번 변경의 본질은 다음 한 문장으로 정리한다.

**ClaudeTool이 잘하는 것은 계속 ClaudeTool에 맡기되, 그 내부에 묻려 있는 도구, 세션 메타, 컨텍스트 조립, 실행 흐름의 통제권을 Aimbase 상위 레이어로 끌어올린다.**

즉, 이번 변경은 다음의 전환이다.

- ClaudeTool 실행 래퍼 → 도메인 앱용 AI 운영 레이어
- 하위 런타임 의존 구조 → 상위 통제 구조
- 단일 솔루션용 구현 → 반복 이식 가능한 공통 기반

# 5. 최종 목표

## 5.1 제품 목표

Aimbase가 AXOPM, 입찰 시스템, WMS, OMS, Mall 등 다양한 도메인 앱에 AI를 빠르고 일관되게 붙일 수 있는 공통 운영 레이어가 되도록 한다.

## 5.2 기술 목표

1. ClaudeTool의 정액제/자율성/고난도 작업 감각을 유지한다.
2. 반복 가능하고 통제 가능한 작업은 Aimbase Native Layer로 이동한다.
3. Session Meta와 Context Assembly는 Aimbase가 직접 장악한다.
4. Runtime 선택은 하위 모듈의 암묵적 독식이 아니라 상위 정책/설정 기반으로 관리한다.
5. 도메인 앱별 기본 recipe, tool allowlist, runtime, MCP 연결 구조를 표준화한다.

## 5.3 이번 단계의 전략 문장

**도구 → 세션 메타 → 컨텍스트 조립 → 자율 런타임 재배치** 순서로, 1단계 안에서 계층 순서대로 압축 실행한다.

여기서 '점진적'은 긴 시간에 걸친 장기 로드맵이 아니라, **구현 순서가 있다는 뜻**이다.

# 8. 상세 기획서

## 8.1 Tool Contract + Workspace Guard

### 목적
기존 ToolExecutor 기반을 깨지 않으면서, Tool 단위를 상위 통제 가능한 계약 객체로 승격한다.

### 설계 원칙
- 기존 ToolExecutor는 하위호환 유지
- EnhancedToolExecutor를 통해 확장
- 모든 네이티브 도구는 contract meta를 가진다
- 입력 검증은 execute 이전에 수행한다
- Workspace Policy는 Tool 실행 전 공통 검증 레이어로 적용한다

### 필수 필드
ToolContractMeta는 최소 다음 정보를 가진다.

- id
- version
- scope
- permissionLevel
- approvalRequired
- readOnly
- destructive
- concurrencySafe
- retryPolicy
- tags
- capabilities

### Workspace Guard 목표
단순 path traversal 방어가 아니라 아래를 포함한다.

- allowed root 제한
- denied path 제한
- 확장자 allow/deny
- max file size 제한
- binary 차단
- secret pattern 탐지

### 완료 기준
- Tool 목록 조회 시 contract meta와 capability가 보인다
- Tool validate / execute 분리 호출 가능
- Workspace Policy 위반 시 실행 전 차단된다

---

## 8.2 Native Tool Layer

### 목표
ClaudeTool 내부의 탐색/읽기/수정 준비 일부를 Aimbase Native Layer로 끌어온다.

### P0 Tool 목표
#### FileReadTool
파일 내용을 제한된 범위로 읽고, 바이너리/메타 처리를 수행한다.

#### GlobTool
파일 집합을 패턴 기반으로 찾는다.

#### GrepTool
텍스트/멀티라인 패턴을 검색한다.

#### WorkspaceSnapshotTool
워크스페이스 구조, 주요 파일군, 최근 변경, git 상태를 요약한다.

#### PathInfoTool
읽기 전 파일/디렉토리의 존재/타입/크기/mtime 등을 확인한다.

#### StructuredSearchTool
class/function/config_key/entity_field/json_path 중심의 구조 검색을 지원한다.

### P1 Tool 목표
#### DocumentSectionReadTool
문서의 특정 섹션, anchor, line range를 선택적으로 읽는다.

#### SafeEditTool
즉시 수정이 아니라 diff/patch 생성 중심으로 동작한다.

#### PatchApplyTool
승인 이후 patch를 적용한다.

### 설계 원칙
- 읽기 도구는 readOnly=true
- 수정 도구는 dryRun 기본
- PatchApply는 승인 연계 필수
- SafeEdit와 Apply를 분리해 거버넌스를 삽입한다

### 완료 기준
- 워크플로우에서 ClaudeTool 없이 읽기/검색/문서 섹션 읽기까지 가능
- SafeEdit는 patch 제안까지만 하고, 실제 변경은 승인 후 적용
- 모든 도구 실행이 lineage와 audit에 남는다

## 8.3 Session Meta Layer

### 목표
세션을 단순 채팅 이력 저장소가 아니라 **작업 문맥 저장소**로 승격한다.

### 추가 메타
- scope_type
- runtime_kind
- workspace_ref
- persistent_session
- summary_version
- context_recipe_id
- last_tool_chain
- app_id
- project_id

### 추가 로그 구조
Tool execution log를 통해 아래를 남긴다.

- 어떤 세션에서
- 어떤 런타임 문맥으로
- 어떤 도구를
- 어떤 입력 요약으로 실행하고
- 어떤 출력/side effect가 나왔는지

### 완료 기준
- 세션 상세에서 tool lineage가 조회 가능
- 세션 단위로 runtime_kind와 recipe가 추적 가능
- app/project 단위의 session grouping이 가능

## 8.4 Context Assembly Layer

### 목표
문맥을 하위 런타임이 암묵적으로 조립하는 구조에서 벗어나, Aimbase가 상위에서 recipe 기반으로 조립한다.

### 컨텍스트 소스
- system_policy
- session_summary
- recent_conversation
- previous_step
- native_tool_results
- mcp_context
- rag
- domain_entity_context
- user_input
- runtime_format

### 핵심 기능
- recipe 기반 layer 정의
- totalTokenBudget
- priority
- evictable
- freshnessMinutes
- deduplication

### 원칙
1. explicit recipe > session meta > domain default > built-in default 순으로 적용
2. budget 초과 시 priority 낮은 evictable layer부터 축소
3. stale layer는 skip
4. recent conversation과 native tool result 중복은 제거

### 완료 기준
- preview API로 assembled messages와 토큰 분포를 미리 볼 수 있다
- recipe가 없으면 기존과 동일한 default assembly로 동작한다
- 리팩터링 후 기존 chat 기본 동작이 깨지지 않는다

---

## 8.5 Runtime Adapter Layer

### 목표
ClaudeTool을 포함한 하위 실행기를 상위 통제 아래 재배치한다.

### 런타임 종류
- ClaudeCodeRuntimeAdapter
- LlmApiRuntimeAdapter
- McpOnlyRuntimeAdapter

### ClaudeTool 역할
ClaudeTool은 다음 역할에 집중한다.

- 긴 문맥 자율 탐색
- 복합 리팩터링
- 로컬 워크스페이스 기반 코딩
- 다중 파일 추론
- 세션 지속이 중요한 작업

### ClaudeTool Mode
- STATELESS
- PERSISTENT

기본값은 STATELESS이며, projectId + sessionId가 있는 경우 PERSISTENT를 허용한다.

### 선택 정책
workflow step에서는 runtime을 명시할 수 있어야 하며, auto 선택 시 selectionReason을 로그에 남긴다.

### 완료 기준
- 특정 step이 왜 ClaudeTool인지, 왜 llm_api인지 설명 가능
- runtime adapter를 추가해도 상위 workflow/config 구조는 유지
- 기존 LLM 호출 경로는 runtime opt-in이 없을 경우 유지된다

## 8.6 Domain 연결고리

### 목표
AXOPM에서 검증한 구조를 다른 도메인 솔루션에 빠르게 이식할 수 있게 한다.

### DomainAppConfig 필드
- domain_app
- default_context_recipe_id
- default_tool_allowlist
- default_tool_denylist
- default_policy_preset
- default_session_scope
- default_runtime
- mcp_server_ids
- config

### 적용 원칙
새 세션 생성 또는 workflow 실행 시 app/domain 기준으로 기본값을 자동 적용한다.

### 완료 기준
- AXOPM이 첫 번째 대표 domain config로 동작한다
- 다른 앱으로 확장 시 도메인별 기본값을 별도 하드코딩하지 않는다

# 9. 선행 개발 설계서

## 9.1 개발 순서 원칙

구현은 아래 순서로 간다.

1. Tool Contract + Workspace Guard
2. Native Tool P0
3. Session Meta
4. Context Assembly
5. Runtime Adapter
6. Domain 연결고리
7. Native Tool P1
8. FE 통합 고도화

이 순서는 '장기 로드맵'이 아니라 **재작업 방지용 계층 순서**다.

## 9.2 Must / Should / Could

### Must
- EnhancedToolExecutor
- WorkspacePolicyEngine
- Tool filtering
- FileRead / Glob / Grep / WorkspaceSnapshot / PathInfo / StructuredSearch
- session meta 확장
- tool execution log
- ContextAssemblyEngine 기본형
- RuntimeAdapter / RuntimeRegistry
- ClaudeTool stateless/persistent 분리

### Should
- DomainAppConfig
- preview API
- FE context visualization 기본형
- DocumentSectionReadTool

### Could
- SafeEdit / PatchApply 고도화
- advanced context diff 시각화
- selection reason 대시보드화

## 9.3 주요 리스크

### 리스크 1. Context 리팩터링 중 기존 동작 깨짐
대응:
- recipe 없으면 default assembly 유지
- 기존 chat 테스트 통과 필수

### 리스크 2. Runtime 추상화가 기존 경로를 방해함
대응:
- runtime opt-in 기본
- runtime 명시 없으면 기존 경로 유지

### 리스크 3. StructuredSearch 범위 과대
대응:
- 1차는 Java class/function + YAML/properties + JSON 일부만 지원

### 리스크 4. WorkspaceSnapshot 환경 의존성
대응:
- git unavailable fallback
- large repo timeout
- recent changes limit 적용

### 리스크 5. PatchApply 충돌/만료 문제
대응:
- patch 상태관리
- apply 전 원본 비교
- conflict 시 reject
- expiration 처리

## 9.4 하위호환 원칙

1. ToolExecutor 기존 경로 유지
2. Context recipe가 없으면 기존 orchestration 유지
3. runtime 필드가 없으면 기존 chat/adapter 호출 유지
4. 기존 built-in tool은 bridge로 동작

## 9.5 테스트 원칙

### Unit
- 각 Native Tool 정상/에러/경계
- WorkspacePolicyEngine
- Context budgeting/priority/freshness/dedup
- Runtime selection

### Integration
- ToolRegistry execute bridge
- OrchestratorEngine default assembly 하위호환
- Session meta read/write
- Domain config 자동 적용

### E2E
- workflow studio에서 native tool 실행
- tool result → next step context 시각화
- recipe preview
- ClaudeTool runtime mode 분기
- SafeEdit → PatchApply → approval flow

# 10. 개발자에게 전달할 최종 문장

이번 CR의 본질은 OpenClaude 포팅이 아니다.

이번 CR의 본질은 **ClaudeTool(= Claude CLI 런타임)의 장점을 유지하면서, 그 내부에 묻려 있는 도구·세션·컨텍스트·실행 흐름의 일부를 Aimbase 상위 레이어로 끌어올려, 도메인 앱에 AI를 반복 가능하게 이식하는 공통 운영 구조를 만드는 것**이다.

개발팀은 세부 구현 시 범용 툴 경쟁이나 Claude 복제를 목표로 잡지 말고, 아래 기준으로 판단해야 한다.

1. 이 기능이 Aimbase 상위 통제권을 강화하는가
2. 이 기능이 도메인 앱 재사용성을 높이는가
3. 이 기능이 ClaudeTool에 묶이지 않고 상위 자산이 되는가
4. 이 기능이 기존 동작을 깨지 않고 도입 가능한가
