## 원문

### 배경: "LLM은 기억하지 않는다 — 컨텍스트 설계가 서비스다"

LLM 서비스의 토큰 비용 절감과 컨텍스트 한계 극복 전략에 대한 논의에서 출발.
핵심 인사이트: LLM은 "기억하는 엔진"이 아니라 "주어진 문맥을 바탕으로 계산하는 엔진"이며,
좋은 AI 서비스의 경쟁력은 모델 자체보다 "무엇을, 언제, 어떤 형태로 모델에 넣을 것인가"에 있다.

### 10가지 핵심 전략

1. **보내는 토큰 자체를 줄이기** — 시스템 프롬프트 최소화, 최근 문맥+핵심 요약만, RAG 청크만, 전처리
2. **비싼 모델을 항상 쓰지 않기** — 단순 분류→저가 모델, 복잡한 추론→고급 모델
3. **호출 횟수 자체를 줄이기** — 한 번에 처리, 중복 호출 방지, 다단계 체인 축소
4. **캐싱하기** — 동일 질문 응답 캐시, 시스템 프롬프트 캐시, 임베딩 결과 캐시
5. **입력 전처리로 LLM에 덜 맡기기** — 정규식, 코드, DB로 기계적 처리
6. **RAG 품질을 높여서 쓸데없는 컨텍스트 감소** — 검색 정확도, 청크 최적화, reranker, top-k 축소
7. **응답 길이 제한** — max_tokens 제한, concise 기본, 요청 시 expanded
8. **대화 메모리 구조 분리** — 최근 대화 + 장기 기억 요약 + 사용자 프로필 + 현재 작업 상태
9. **비동기/지연 실행** — 문서 요약, 태깅, 임베딩, 보고서 초안을 백그라운드에서 미리 생성
10. **재시도 전략 최적화** — 부분 실패만 재시도, 프롬프트 품질로 한 번에 맞게

### Claude Code/Codex 등 에이전트형 툴의 메모리 전략

- 프로젝트 규칙은 작은 파일에 (CLAUDE.md, AGENTS.md)
- 원문은 저장소에 남기고 필요 시 읽음
- 대화가 길어지면 상태 요약으로 압축
- 반복되는 prefix는 캐시
- 큰 문제는 단계적으로 푼다 (탐색→추출→판단→작성)

### 모든 LLM의 공통 한계

1. **컨텍스트 윈도우 한계** — 한 번에 처리 가능한 입력량이 유한
2. **영구 기억 불가** — 매 호출 시 들어온 컨텍스트 안에서만 계산
3. ChatGPT "학습" 기능도 실상은 외부 메모리/기록 계층에 저장 후 필요 시 참조

핵심 표현: "LLM은 두뇌라기보다 CPU에 가깝고, 메모리/디스크/RAG/상태저장이 바깥에 따로 붙어야 한다."

### 식별된 5개 Gap

**Gap 1: 대화 요약/압축** — 컨텍스트 초과 시 오래된 메시지를 그냥 버림. 요약 후 보존 필요.

**Gap 2: Smart Model Routing** — 모든 요청이 동일 모델(Sonnet)로 감. 의도 기반 분기 필요.

**Gap 3: LLM 응답 캐시** — 같은 질문도 매번 새로 호출. Exact + Semantic 캐시 필요.

**Gap 4: 메모리 구조 분리** — 단일 메시지 리스트. 장기/단기/프로필/규칙 계층 분리 필요.

**Gap 5: Fallback Chain** — 모델 장애 시 대응 없음. 자동 전환 + 서킷 브레이커 필요.

### Aimbase 고유 차별점

- Database-per-Tenant 멀티테넌시 (다른 오픈소스에 없음)
- Self-RAG + HyDE (검색 품질 자동 평가 + 가설적 문서 생성)
- SpEL 기반 정책 엔진 (조건식으로 유연한 규칙 평가)
- PII 마스킹 6종 (이메일, 전화, 카드, 주민번호 등)
- 토큰 쿼터 관리 (테넌트별 사용량 제한)
