# CR-029: Aimbase 1단계 고도화 — Native Tool Foundation + Session Meta + Context Assembly + Runtime 재배치

> **v3.0** — OpenClaude 운영 인사이트 반영 + 품질 비교 가능성 설계 (2026-04-06)

## Context

Aimbase가 "ClaudeTool 실행 래퍼"에서 벗어나, 도메인 앱(AXOPM, 입찰, WMS, OMS, Mall)에 AI를 반복 이식하는 **공통 운영 레이어**로 재정렬한다. 핵심 순서: **도구 → 세션 메타 → 컨텍스트 조립 → 런타임 재배치**.

### 프로젝트의 본질 (제품/아키텍처 브리프 기준)

이번 CR은 OpenClaude 포팅이 아니다. Claude 복제도 아니다.
**ClaudeTool 내부에 묻혀 있는 운영 인사이트(도구 연쇄, 세션 지속, 문맥 조립, 실행 흐름)의 일부를 Aimbase 상위 레이어로 끌어올리는 프로젝트**다.

OpenClaude는 "Claude CLI가 왜 자연스럽고 만능적으로 느껴지는지" 그 외곽 레이어의 구조적 힌트를 확인하기 위해 분석했으며, 개별 도구 포팅이 아니라 **운영 패턴의 인사이트**를 추출했다.

### 품질 비교 가능성 원칙 (v3 신규)

> 이번 설계는 구현 이후 ClaudeTool과 품질 비교가 가능해야 한다.

각 계층은 **재현 가능성, 추적 가능성, 비교 가능성**을 전제로 설계한다:

1. **도구 호출 순서** — tool_execution_log에 호출 순서, 입력 요약, 출력 요약, 소요 시간이 남아야 한다
2. **세션 메타** — 어떤 runtime_kind, scope_type, recipe로 실행했는지 추적 가능해야 한다
3. **컨텍스트 조립 결과** — assembled messages의 토큰 분포, 포함/제외된 source, budget eviction 이력이 남아야 한다
4. **런타임 선택 이유** — selectionReason이 로그에 명시되어야 한다
5. **실행 lineage** — 동일 과제를 ClaudeTool과 Aimbase 워크플로우로 각각 재실행해 비교할 수 있어야 한다

### OpenClaude 분석에서 추출한 핵심 운영 인사이트 (v3 신규)

| 인사이트 | Claude CLI가 하는 것 | CR-029 반영 |
|----------|---------------------|-------------|
| **스트리밍 병렬 도구 실행** | 모델 응답 중 도구 시작, concurrencySafe 분할 | ToolContractMeta.concurrencySafe + ToolCallHandler 병렬 분기 |
| **결과 크기 3단계 관리** | per-tool → per-message(3MB) → 파일 저장+preview | ToolResult.summary + tool_execution_log에 full output 보관 |
| **ContentReplacementState** | 대용량 결과를 frozen/fresh로 분류, 캐시 안정성 우선 | context assembly 시 tool result 캐시 전략 |
| **하이브리드 토큰 추정** | API 실측 + char/4 결합 | ContextAssemblyEngine에 하이브리드 추정 도입 |
| **Effective Window** | 전체 - 출력예약(20k) = 유효 입력, 모델별 계산 | recipe.totalTokenBudget을 모델별 유효 윈도우로 자동 조정 |
| **Auto-compact circuit breaker** | 연속 3회 실패 시 압축 중단 | ContextAssemblyEngine에 circuit breaker 내장 |
| **다단계 압축** | snip → microcompact → autocompact | 경량 필터 → 요약 2단계 |
| **Permission 6소스 계층** | managed→user→project→local→flag→session | WorkspacePolicyEngine 계층화 |
| **Denial tracking** | 연속 3회/총 20회 거부 시 폴백 | 자동 실행 모드의 안전장치 |
| **parentSessionId 체인** | 세션 계보 추적, 서브에이전트 격리 | Session Meta에 parent_session_id 추가 |

## Phase 1. Tool Contract + Workspace Guard (P0)

### 1-1. 새 인터페이스 & 레코드 (기존 호환 유지)

**패키지**: `com.platform.tool`

| 클래스 | 역할 |
|--------|------|
| `ToolContext` (record) | tenantId, appId, projectId, sessionId, workflowRunId, actorUserId, permissionLevel(enum READ_ONLY/RESTRICTED_WRITE/FULL), approvalState(enum NOT_REQUIRED/PENDING/APPROVED/DENIED), workspacePath, dryRun |
| `ToolResult` (record) | success, output(Object), summary(String), artifacts(List\<ToolArtifact\>), sideEffects(List\<String\>), auditPayload(Map), nextContextHint, durationMs |
| `ToolArtifact` (record) | type, ref, data |
| `ToolContractMeta` (record) | id, version, scope(enum BUILTIN/MCP/NATIVE/EXTERNAL), permissionLevel, approvalRequired, readOnly, destructive, concurrencySafe, retryPolicy(record maxRetries/backoffMs), tags(List), capabilities(List) |
| `EnhancedToolExecutor` (interface extends ToolExecutor) | `getContractMeta()`, `execute(Map, ToolContext) -> ToolResult`, `validateInput(Map, ToolContext) -> ValidationResult`, default bridge로 기존 execute(Map)->String 유지 |
| `ValidationResult` (record) | valid(boolean), message(String), errorCode(int) |

**핵심 원칙**: `ToolExecutor`는 건드리지 않는다. `EnhancedToolExecutor`를 새로 만들어 기존 빌트인(Calculator, Notification 등)은 그대로 동작.

### 1-2. Workspace Guard Layer

**패키지**: `com.platform.tool.workspace`

| 클래스 | 역할 |
|--------|------|
| `WorkspaceResolver` (@Component) | 테넌트/앱/프로젝트별 workspace 루트 해석, 경로 상대화 |
| `WorkspacePolicy` (record) | allowedRoots(List), deniedPaths(List), allowedExtensions(List), deniedExtensions(List), maxFileSize(long), denyBinary(boolean), secretPatterns(List\<Pattern\>) |
| `WorkspacePolicyEngine` (@Component) | WorkspacePolicy 평가 + path traversal 방어 + secrets 차단 |

**WorkspacePolicy 적용 계층**:
- 글로벌 기본 정책 (application.yml)
- 테넌트별 정책 오버라이드 (global_config 또는 workspace_policies 테이블)
- 프로젝트별 정책 오버라이드 (project_resources 확장)

### 1-3. Tool Capability Filtering

기존 `ToolFilterContext`를 확장하여 capability 기반 필터링 추가:

```java
public record ToolFilterContext(
    List<String> allowedTools,      // 기존
    List<String> excludeTools,      // 기존
    List<String> tags,              // 기존
    // --- 신규 ---
    List<String> requiredCapabilities,  // e.g., ["read", "search"]
    PermissionLevel maxPermission,      // 이 세션/워크플로우의 최대 허용 수준
    boolean readOnlyMode                // true면 readOnly 도구만 허용
)
```

**적용 계층** (우선순위 높은 순):
1. 워크플로우 step 설정 (step config에 tool_filter)
2. 세션 설정 (session meta의 scope_type 기반)
3. 앱/도메인 기본값 (domain_app별 default tool allowlist)
4. 테넌트 정책 (policies 테이블의 TOOL_FILTER 타입)

### 1-4. ToolRegistry 확장

파일: `backend/platform-core/src/main/java/com/platform/tool/ToolRegistry.java`

- `execute(ToolCall, ToolContext)` 메서드 추가 — executor가 `EnhancedToolExecutor`이면 validateInput → 새 execute 호출, 아니면 기존 bridge
- 실행 전: `WorkspacePolicyEngine.validate()` 호출
- 실행 후: `ToolResult.auditPayload`를 `AuditLogger`에 전달
- `getToolDefs(ToolFilterContext)` 확장: capability/permission 필터링 추가, `EnhancedToolExecutor`이면 `ToolContractMeta`도 함께 반환

### 1-5. ToolCallHandler 확장 (v3 보강)

파일: `backend/platform-core/src/main/java/com/platform/tool/ToolCallHandler.java`

- `executeLoop()` 시그니처에 `ToolContext` 파라미터 추가
- **v3: 병렬 실행** — 한 턴에 여러 tool_use가 오면 `concurrencySafe=true`인 도구끼리 병렬, 아니면 순차 (OpenClaude partition 패턴)
- 각 도구 실행 결과를 `ToolExecutionLogEntity`로 DB 저장 (lineage)
- **v3: 호출 순서 번호(sequence)** — 동일 턴 내 실행 순서를 명시적으로 기록 (비교 가능성)

### 1-6. Tool API 엔드포인트

기존 `ToolController`에 추가:

```
GET  /api/v1/tools                           — 전체 도구 목록 (contract 포함, capability 필터)
GET  /api/v1/tools/{toolName}/contract       — Contract 상세
POST /api/v1/tools/{toolName}/execute        — 직접 실행 (input, dryRun?, workspacePath?)
POST /api/v1/tools/{toolName}/validate       — 입력 검증만 (dry validation)
```

### 4-2. ContextAssemblyEngine (v3 — 비교 가능성 + 운영 인사이트 반영)

**패키지**: `com.platform.context`

```java
@Component
public class ContextAssemblyEngine {

    public AssemblyResult assemble(
        String sessionId, String recipeId, ChatRequest request, StepContext stepContext);

    // Recipe 해석: explicit ID > session meta > domain default > built-in default
    private ContextRecipeEntity resolveRecipe(String recipeId, SessionMeta meta);

    // --- v2: Budget & Scoring ---
    private List<AssembledLayer> applyBudget(List<AssembledLayer> layers, int totalTokenBudget);
    private boolean isStale(AssembledLayer layer, ContextSourceConfig config);
    private List<UnifiedMessage> deduplicate(List<UnifiedMessage> messages);

    // --- v3 신규: 하이브리드 토큰 추정 ---
    /**
     * OpenClaude 패턴: 마지막 API 응답의 실측 토큰 + 이후 메시지는 char/4.
     * lastApiUsage가 있으면 정확도 높은 추정, 없으면 전체 char/4 fallback.
     */
    private int estimateTokens(List<UnifiedMessage> messages, TokenUsage lastApiUsage);

    // --- v3 신규: Effective Window 자동 계산 ---
    /**
     * 모델별 유효 윈도우 = contextWindow - maxOutputTokens(20k).
     * recipe.totalTokenBudget이 null이면 이 값을 자동 사용.
     */
    private int getEffectiveWindow(String model);

    // --- v3 신규: Auto-compact circuit breaker ---
    /**
     * 연속 3회 압축 실패 시 압축 중단 (무한 루프 방지).
     */
    private boolean shouldSkipCompact(String sessionId);
}

// v3: AssemblyResult — 비교 가능성을 위한 구조화된 결과
record AssemblyResult(
    List<UnifiedMessage> messages,           // 최종 조립 결과
    AssemblyTrace trace                      // 조립 과정 추적 (비교용)
) {}

record AssemblyTrace(
    String recipeId,                          // 사용된 recipe
    String resolveReason,                     // recipe 선택 이유
    List<AssembledLayer> includedLayers,       // 포함된 layer + 각 토큰 수
    List<String> evictedLayers,               // budget 초과로 제거된 layer
    List<String> staleLayers,                 // freshness 만료로 skip된 layer
    int totalEstimatedTokens,                 // 최종 추정 토큰
    int effectiveWindow,                      // 모델별 유효 윈도우
    long assemblyDurationMs                   // 조립 소요 시간
) {}

record AssembledLayer(
    String source,
    List<UnifiedMessage> messages,
    int estimatedTokens,
    int priority,
    boolean evictable
) {}
```

### 5-4. ClaudeTool Mode 분리

```java
enum ClaudeCodeRuntimeMode { STATELESS, PERSISTENT }
```

- `workflowRunId != null` → STATELESS
- `projectId != null && sessionId != null` → PERSISTENT
- 기본값: STATELESS (application.yml: `claude-code.default-runtime-mode: stateless`)

ClaudeCodeTool → `EnhancedToolExecutor` 구현:
- `getContractMeta()`: id=`builtin:claude_code`, readOnly=false, scope=NATIVE
- `execute(Map, ToolContext)`: mode 결정 → pre-execution log → 기존 실행 → post-execution log → ToolResult
- Audit linkage: workflowRunId, sessionId를 auditPayload에 포함
