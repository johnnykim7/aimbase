# LLM_CALL 토큰 초과 자동 처리 전략 (CR-028)

- **요청자**: sykim
- **요청일**: 2026-03-28
- **맥락**: AXOPM OPM 구조 자동 생성 테스트 중 발견

## 1. 문제 상황 + 목표

### 문제
- OPM 구조 자동 생성 워크플로우에서 LLM 응답이 `max_tokens=4096`에서 잘림
- 잘린 structured output이 빈 `{}`로 반환 → 소비앱에 빈 화면 표시
- 소비앱 개발자가 토큰 전략을 알아야 하는 구조 = 플랫폼 설계 실패

### 목표
- 소비앱은 토큰을 몰라도 됨 — 플랫폼이 자동 처리
- 어떤 크기의 출력이든 완전한 structured output 보장
- 비용 효율 (최소 토큰부터 시작, 필요할 때만 분할)
- 실패 시 원인이 명확한 에러 메시지

## 2. 아키텍처 선택 (결정 사항)

**에스컬레이션 1회 → 실패 시 자동분할(Auto-Split)** 2단계 전략 채택

```
1차: 일반 호출 (4096) → 성공 시 바로 반환
2차: 에스컬레이션 (8192) → 성공 시 반환
3차: 자동분할 모드 진입
  3-a: 분할 계획 호출 → 파트 목록 획득
  3-b: 파트별 실행 (각 4096)
  3-c: 취합 호출 → 최종 결과 반환
```

| 단계 | 역할 | 대상 케이스 |
|------|------|-----------|
| 에스컬레이션 | 살짝 넘는 경우 빠르게 해결 | output 4097~8192 토큰 |
| 자동분할 | 크게 넘는 경우 확실하게 해결 | output 8192+ 토큰 |

## 3. 핵심 파라미터 (확정)

```
initial_max_tokens: 4096          # 첫 시도 (비용 효율)
escalation_max_tokens: 8192       # 에스컬레이션 1회
split_part_max_tokens: 4096       # 파트별 호출 토큰
split_plan_max_tokens: 4096       # 분할 계획 호출 토큰
split_merge_max_tokens: 4096      # 취합 호출 토큰
split_max_parts: 10               # 최대 분할 파트 수 (비용 폭발 방지)
config_ceiling_respected: true    # step config max_tokens는 상한선으로 존중
```

## 4. 처리 흐름 (단계별)

### Phase 1: 일반 호출
```
1. step config에서 max_tokens 읽기 (없으면 null)
2. 첫 시도: max_tokens = min(4096, config상한) 으로 LLM 호출
3. finishReason 확인
   - END/TOOL_USE → 성공, 결과 반환
   - MAX_TOKENS → Phase 2로
```

### Phase 2: 에스컬레이션 (1회)
```
4. max_tokens = min(8192, config상한) 으로 재시도
5. finishReason 확인
   - END/TOOL_USE → 성공, 결과 반환
   - MAX_TOKENS → Phase 3로
6. log.warn("step '{}': 에스컬레이션 8192로 재시도", stepId)
```

### Phase 3: 자동분할

#### 3-a. 분할 계획 호출
```
7. 분할 계획 프롬프트 구성:
   - system: "당신은 작업 분할 전문가입니다."
   - prompt: "다음 작업의 출력이 너무 큽니다.
     독립적으로 생성 가능한 N개 파트로 분할하세요.
     [원본 system prompt 요약]
     [원본 prompt]
     [response_schema]"
   - response_schema: {parts: [{part_number, scope, description}]}
8. LLM 호출 (4096) → 파트 목록 획득
9. 파트 수 검증: 10개 초과 시 10개로 제한
```

#### 3-b. 파트별 실행
```
10. 각 파트에 대해:
    - system: 원본 system prompt
    - prompt: "전체 작업 중 파트 N만 생성하세요.
      범위: [scope]
      [원본 prompt]"
    - response_schema: 없음 (자유 형식 JSON)
11. LLM 호출 (4096) → 파트별 JSON 조각 수집
12. log.info("step '{}': 파트 {}/{} 완료", stepId, i, total)
```

#### 3-c. 취합 호출
```
13. 취합 프롬프트 구성:
    - system: "JSON 조각을 하나의 완전한 구조로 병합하세요."
    - prompt: "다음 조각들을 이 스키마에 맞게 합치세요:
      [response_schema]
      조각들:
      [파트1 결과]
      [파트2 결과]
      ..."
    - response_schema: 원본 response_schema
14. LLM 호출 (4096) → 최종 structured_data
15. log.info("step '{}': 자동분할 완료 ({}개 파트)", stepId, total)
16. 결과 반환 (소비앱에는 일반 호출과 동일한 형태)
```

## 5. 예외 처리 전략

| 상황 | 처리 |
|------|------|
| Phase 1 MAX_TOKENS | Phase 2 에스컬레이션 |
| Phase 2 MAX_TOKENS | Phase 3 자동분할 |
| 분할 계획 호출 실패 | RuntimeException — "자동분할 계획 생성 실패" |
| 파트 실행 중 1개 실패 | 해당 파트만 1회 재시도, 재실패 시 전체 실패 |
| 파트 실행 중 MAX_TOKENS | 해당 파트의 scope가 너무 큼 — 전체 실패 + "수동 스텝 분할 권고" |
| 취합 호출 MAX_TOKENS | 4096 → 8192 에스컬레이션 후에도 실패 시 — 전체 실패 + "수동 스텝 분할 권고" |
| 취합 결과가 schema 불일치 | 전체 실패 + "자동분할 취합 실패" |
| 네트워크/API 에러 | 에스컬레이션/분할 루프 밖으로 throw → executeWithRetry 위임 |
| response_schema 없는 경우 | 자동분할 불가 → 에스컬레이션까지만 시도 후 실패 |

## 6. 품질 기준

| 지표 | 기준 | 측정 방법 |
|------|------|----------|
| structured_data 완전성 | 잘림으로 인한 빈 `{}` 반환 0건 | step_results 확인 |
| 비용 효율 | 4096 이내 응답은 1회 호출로 완료 | 에스컬레이션/분할 미발생 확인 |
| 자동분할 성공률 | schema 있는 호출 95%+ 성공 | 분할 후 취합 결과 schema 검증 |
| 소비앱 호환 | 기존 워크플로우 동작 변화 없음 | 기존 워크플로우 실행 테스트 |
| 투명성 | 소비앱은 분할 여부를 모름 | 반환 구조 동일 확인 |

## 발견 경위

AXOPM 소비앱의 "구조 설계 Builder" 화면에서 AI 빌더로 OPM 구조를 자동 생성하는 기능 테스트.
"WMS 시스템에 대해 테스트를 하려고 합니다. 이에 대한 AI 6단계 전략을 세워주세요..." 입력 후
"AI 분석 요약" 섹션 아래에 6단계 OPM 구조가 표시되어야 하나 빈 화면.

### 워크플로우 실행 로그

```json
// Step 1: RAG 검색 — 빈 결과
{"output": "{\"query\": \"WMS 시스템에 대해 테스트를...\", \"results\": []}"}

// Step 2: OPM 구조 생성 — output_tokens 4096 도달, structured_data 빈 객체
{"model": "claude-sonnet-4-20250514", "output": "", "output_tokens": 4096, "structured_data": {}}

// Step 3: 검증 — 빈 구조 감지
{"structured_data": {"valid": false, "issues": ["생성된 구조가 비어있음 ({})"]}}
```

### 원인 분석

- AnthropicAdapter max_tokens 기본값 4096 고정
- LlmCallStepExecutor가 step config의 max_tokens를 읽지 않음
- CR-017 증거 요건(evidence_requirements) 추가로 OPM 구조 출력량 급증
- 4096 토큰에서 tool_use JSON 잘림 → 파싱 실패 → structured_data: {}

### 요청자 핵심 요구

1. "소비앱에서 이런걸 일일이 할 수 없잖아요" — 플랫폼이 자동 처리해야 함
2. "토큰 최저로 분할하게 답변 달라고 하고, 그 응답을 나중에 자동 취합해서 소비앱에 전달하면 되잖아요"
3. "JSON 등이 분할되어도 나중에 결국 다 취합하는 거죠?" — 소비앱은 완성된 JSON 하나만 받음
