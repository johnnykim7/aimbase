## 원문

### 배경

AI 튜터 백엔드 플랫폼 비교 분석에서 Aimbase를 포함한 4개 플랫폼(Aimbase, Open WebUI, n8n, Dify)을 평가한 결과, Aimbase RAG 파이프라인이 A 등급으로 평가됨. 사용자가 "A++로 격상시키려면 어떤 걸 더 구현해야 하나요?" 질문.

### 핵심 차이점 분석

- **Open WebUI** — "채팅 인터페이스"이지 "앱 백엔드"가 아님. 데모/프로토타입에는 좋으나, 커스텀 React 프론트와 결합하려면 아키텍처 불일치.
- **n8n** — "워크플로우 도구"이지 "앱 백엔드"가 아님. 복습 알림 스케줄링, PDF 자동 인제스션 같은 보조 자동화로는 최강이나, 채팅 세션/사용자 인증/DB 스키마 관리는 별도 필요.
- **Dify** — API-first 설계로 커스텀 React 프론트 연동에 가장 적합. RAG+채팅+TTS/STT 올인원. 단, SM-2 같은 도메인 DB 로직은 Dify 밖에서 관리해야 함.
- **Aimbase** — 전체 코드 통제권 + 멀티테넌시 + 정책 엔진이 이미 있음. TTS/STT만 미구현.

### 추천 옵션 (사용자 선택: A)

| 옵션 | 구성 | 장점 | 단점 |
|------|------|------|------|
| **A. Aimbase 단독** | Aimbase + TTS/STT 추가 개발 | 풀 커스텀, 멀티테넌시, 코드 통제 | 전부 자체 유지보수 |
| B. Dify + 보조 백엔드 | Dify(AI 엔진) + 경량 백엔드(SM-2/학습이력) | 빠른 MVP, AI 기능 즉시 사용 | 두 백엔드 운영, Dify 의존 |
| C. Aimbase + n8n | Aimbase(코어) + n8n(알림/자동화) | 각각 강점 활용 | 통합 복잡도 증가 |

### 사용자 의견

> "aimbase도 상대방거 가져다가 사용하면 손색이 없네요?"

→ Aimbase가 이미 채팅/RAG/프롬프트/워크플로우/멀티테넌시/정책엔진까지 갖추고 있어서, 부족한 건 TTS/STT 정도뿐이고 그것도 OpenAI TTS/Whisper API 연동이면 해결됨.

### RAG A++ 격상 요구사항

사용자 질문: "위 점수표에서 RAG라든가 A인데, A++로 격상시키려면 어떤걸 더 구현해야하나요?"

분석 결과 도출된 10개 미구현 영역 (우선순위순):

#### 1. Contextual Retrieval (임팩트: ★★★★★)
Anthropic이 발표한 기법. 각 청크에 **문서 전체 맥락 요약**을 prepend하고 임베딩 → 검색 정확도 49% 향상.
- 현재: 청크가 원문 그대로 임베딩됨
- 필요: 청킹 시 LLM으로 `"이 청크는 [문서제목]의 [섹션]에서 [맥락]을 다루는 내용입니다"` 프리픽스 자동 생성

#### 2. Parent-Child (계층적) 청크 검색 (임팩트: ★★★★★)
- 현재: 검색된 청크 = 반환 청크 (동일)
- 필요: **작은 청크로 매칭** → **부모(큰) 청크를 반환**하여 충분한 컨텍스트 제공
- Dify/LlamaIndex의 핵심 차별 기능

#### 3. Query Transformation 고도화 (임팩트: ★★★★)
- **HyDE**: 가상 답변 생성 → 그걸로 검색 (빈약한 질문도 좋은 결과)
- **Multi-Query**: 하나의 질문을 3~5개 변형 질문으로 확장 → 합집합 검색
- **Step-back Prompting**: 구체적 질문 → 상위 개념 질문으로 변환
- 현재: `transform_query` API만 존재, 구체 구현 부족

#### 4. RAG 평가 파이프라인 (RAGAS) (임팩트: ★★★★)
- **Faithfulness**: 답변이 검색된 컨텍스트에 충실한가
- **Context Relevancy**: 검색된 청크가 질문에 관련있는가
- **Answer Relevancy**: 답변이 질문에 답하는가
- **Context Recall/Precision**: ground truth 대비 검색 품질
- 현재: 평가 메트릭 전무 → 품질 개선 루프 불가

#### 5. 고급 문서 처리 (임팩트: ★★★★)
- **테이블 구조 추출**: PDF 내 표를 Markdown 테이블로 변환
- **레이아웃 인식 파싱**: 2단 레이아웃, 캡션, 헤더/푸터 구분
- **OCR**: 스캔 PDF 지원 (Tesseract/PaddleOCR)
- **이미지 이해**: 도표/차트를 텍스트 설명으로 변환 (멀티모달)
- 현재: Tika 기본 텍스트 추출만

#### 6. 시맨틱 캐시 (임팩트: ★★★)
- 유사 질문 감지 → 이전 검색/답변 결과 재사용
- Redis에 질문 임베딩 저장 → 코사인 유사도 > 0.95면 캐시 히트
- 비용 절감 + 응답 속도 대폭 향상

#### 7. Graph RAG (임팩트: ★★★) — 향후 고려
- 문서에서 엔티티/관계 추출 → 지식 그래프 구축
- Vector 검색 + Graph traversal 결합
- "A와 B의 관계는?" 같은 멀티홉 질문에 강력
- Microsoft GraphRAG 패턴

#### 8. 증분 인제스션 + 변경 감지 (임팩트: ★★★)
- 현재: sync 시 전체 삭제 후 재임베딩
- 필요: 문서 해시 비교 → 변경분만 재처리
- 대용량 소스에서 비용/시간 절감

#### 9. 출처 추적 (Citation) (임팩트: ★★★)
- LLM 답변에 `[1]`, `[2]` 인라인 인용 번호 삽입
- 각 인용이 어떤 문서의 몇 페이지인지 링크
- Dify/Perplexity 수준의 사용자 신뢰도 확보

#### 10. 검색 분석 대시보드 (임팩트: ★★) — 향후 고려
- 질문별 검색 적중률, 평균 유사도, 빈출 질문 등
- 어떤 청크가 자주 검색되는지, 사용 안 되는 문서는 무엇인지
- RAG 품질 모니터링 루프

### 구현 로드맵 (합의)

| 단계 | 기능 | 예상 공수 | A++ 기여도 |
|------|------|----------|-----------|
| **Phase 1** | Contextual Retrieval + Parent-Child 청크 | 3~4일 | +0.5등급 |
| **Phase 2** | Query Transformation (HyDE, Multi-Query) | 2~3일 | +0.3등급 |
| **Phase 3** | RAGAS 평가 파이프라인 | 2~3일 | +0.3등급 |
| **Phase 4** | 고급 문서 처리 (테이블/OCR) | 3~4일 | +0.2등급 |
| **Phase 5** | 시맨틱 캐시 + 증분 인제스션 + Citation | 3~4일 | +0.2등급 |

**Phase 1~3만 구현하면 A+ 확보**, Phase 4~5까지 하면 **A++ (Dify 대비 우위)**.

AI 튜터 맥락에서 특히 중요한 건 **Contextual Retrieval**(교재 맥락 보존)과 **Citation**(학생에게 출처 제시).
