### [대화 1] 1단계 기획서

Aimbase 1단계 고도화 기획서
부제: 도메인 앱용 AI 운영 레이어로서의 Aimbase 재정렬

#### 1. 문서 목적

본 문서는 Aimbase를 범용 AI 제품으로 키우기 위한 문서가 아니다.
본 문서의 목적은 AXOPM, 입찰 시스템, WMS, OMS, Mall 등 도메인 솔루션에 AI 기능을 빠르고 일관되게 이식하기 위한 공통 운영 레이어로 Aimbase를 고도화하는 것이다.

즉, Aimbase 자체를 바로 상용 솔루션으로 판매하는 것이 1차 목적이 아니라:

여러 도메인 앱에 AI를 반복 가능하게 붙이고
모델/런타임이 바뀌어도 상위 구조를 유지하며
정책/감사/승인/세션/워크플로우를 공통화하고
도메인별 MCP/Tool/Entity Context를 표준 방식으로 연결

하기 위한 내재화된 AI 운영 엔진으로 정렬하는 것이 목적이다.

#### 2. 핵심 정의

2.1 Aimbase의 정체성

Aimbase는 범용 챗 제품이 아니다.
Aimbase는 도메인 앱에 붙는 AI 운영 레이어다.

2.2 ClaudeTool의 정체성

ClaudeTool은 Aimbase를 대체하는 제품이 아니다.
ClaudeTool은 하위 런타임이다.

2.3 이번 1단계의 본질

이번 1단계는 Claude CLI를 제거하는 프로젝트가 아니다.
이번 1단계는 Claude CLI의 비용/품질/자율성 이점은 유지하면서, 도구·세션·컨텍스트·통제권을 Aimbase 상위 레이어로 끌어오는 프로젝트다.

#### 3. 왜 지금 이 작업을 하는가

현재 Aimbase는 이미 아래 역량을 보유한다.

워크플로우 생성/실행 구조
LLM_CALL, TOOL_CALL, ACTION, CONDITION, PARALLEL, HUMAN_INPUT 기반 DAG
세션 영구 저장과 히스토리 조회
컨텍스트 요약 및 윈도우 관리
MCP 서버 등록 및 도구 탐색
정책, 승인, 감사, 사용량/비용 관리
RAG 지식소스 업로드/인제스션/검색

반면 현재 ClaudeTool/Claude CLI가 흡수하고 있는 영역이 크다.

파일 읽기/탐색/수정
로컬 워크스페이스 기반 자율 실행
다중 파일 추론
대형 문맥 처리
세션/IDE 연속성 일부

이 상태가 계속되면, Aimbase는 "상위 플랫폼"이 아니라 "ClaudeTool 실행 래퍼"에 가까워질 위험이 있다.
따라서 1단계에서 Aimbase의 본체를 명확히 재정의해야 한다.

#### 4. 1단계의 목표

최종 목표

Aimbase가 도메인 앱용 AI 운영 레이어로서 상위 통제권을 확보한다.

세부 목표
Claude CLI의 정액제/자율성 장점은 유지한다.
반복 가능하고 추적 가능한 작업은 Aimbase 네이티브 Tool로 끌어온다.
세션/컨텍스트/정책/감사/승인은 Aimbase가 주도한다.
AXOPM을 첫 번째 대표 도메인으로 삼아, 이후 다른 솔루션에도 재사용 가능하도록 구조를 설계한다. AXOPM은 이미 Aimbase를 "앱에 AI를 쉽게 붙이는 AI 엔진"으로 설명하고 있고, AXOPM MCP 35개 도구를 통해 프로젝트 관리 전반을 AI에 연결하는 구조를 전제로 한다.

#### 5. 1단계의 명확한 원칙

원칙 1. 장기 점진이 아니라, 1단계 안에서 계층 순서대로 압축 실행한다

이번 작업은 1년짜리 느린 장기 과제가 아니다.
다만 순서는 반드시 지킨다.

실행 순서:

도구
세션 메타
컨텍스트 조립
런타임 일부 흡수

원칙 2. ClaudeTool을 없애지 않는다

ClaudeTool은 1단계에서 제거 대상이 아니다.
고난도 자율 탐색, 다중 파일 추론, 복합 리팩터링, 실제 코딩 흐름은 계속 ClaudeTool이 담당한다.

원칙 3. Aimbase가 가져가야 하는 것은 "도구 목록"이 아니라 "상위 통제권"이다

즉, 내재화의 목표는 기능 복제가 아니라 아래를 Aimbase 자산으로 만드는 데 있다.

툴 계약
실행 로그
승인/정책 연결
세션 메타
컨텍스트 조립
도메인 컨텍스트 결합

원칙 4. 범용 경쟁을 하지 않는다

OpenWebUI, n8n과 같은 범용 제품과 직접 경쟁하려 하지 않는다.
Aimbase는 내부 도메인 솔루션 포트폴리오를 위한 공통 AI 운영층이다.

#### 6. 1단계 범위

6.1 In Scope

A. Native Tool Layer 1차 구축

반드시 구현할 도구:

FileReadTool
GlobTool
GrepTool
SafeEditTool(제한형)
DocumentSectionReadTool
StructuredSearchTool
WorkspaceSnapshotTool

B. Tool Contract 표준화

모든 네이티브 Tool은 아래 메타를 강제한다.

tool id
description
input schema
output schema
permission level
approval requirement
audit payload
retry policy
scope(project/workspace/tenant)
side effect 여부

C. Session Meta Layer 강화

현재 Aimbase는 대화 세션 DB 영구 저장, 히스토리 목록 조회, 대화 요약 생성, 요약 주입 기반 context window 관리 기능을 이미 보유한다. 1단계에서는 이 기능을 ClaudeTool 연계까지 확장한다.

추가 구현 대상:

project/workspace 단위 세션 식별자
runtime type(ClaudeTool/API/MCP) 저장
session mode(stateless / persistent) 저장
tool execution lineage 저장
summarized context snapshot 저장

D. Context Assembly Layer 구축

입력 데이터를 무작정 ClaudeTool에 던지지 않고, Aimbase가 상위에서 컨텍스트를 조립한다.

컨텍스트 원천:

세션 요약
최근 대화
워크플로우 이전 step 출력
Native Tool 결과
MCP 조회 결과
RAG 검색 결과
도메인 엔티티 링크 결과

AXOPM에서는 OPM 6단계 + WBS + Issues + OKR + Risk + Worklog를 하나의 컨텍스트로 제공하는 것이 강점으로 제시되어 있으므로, 이 교차 모듈 통합 컨텍스트를 Context Assembly의 대표 모델로 삼는다.

E. ClaudeTool 재배치

ClaudeTool은 아래 역할에 집중시킨다.

긴 문맥 자율 탐색
복합 리팩터링
로컬 워크스페이스 기반 코딩
다중 파일 추론
세션성/IDE 연속성이 중요한 작업

단, 실행 전/후 메타는 Aimbase에 남긴다.

F. Workflow Studio 반영

기존 워크플로우 스튜디오/TOOL_CALL 구조를 활용해, 네이티브 Tool을 1급 객체로 노출한다. Aimbase는 이미 비주얼 캔버스, 노드 팔레트, 설정 패널, 실행 시각화 구조를 갖추고 있으므로, 여기에 Tool Layer를 더 명시적으로 드러낸다.

6.2 Out of Scope

1단계에서 하지 않을 것:

범용 멀티 에이전트 swarm 엔진
모든 Claude CLI 기능 완전 복제
범용 IDE 제품화
범용 앱 마켓플레이스
외부 판매용 상용 패키징
모든 provider 공통 런타임 완성형 추상화

8.4 ClaudeTool 실행 모드 설계

Mode A. Stateless

용도:

재현 가능한 워크플로우 step
짧은 일회성 분석
결과 검증이 쉬운 작업

특징:

Mode B. Persistent Project Session

용도:

실제 개발/분석 흐름
프로젝트 단위 탐색

특징:

정책:

1단계에서는 두 모드를 모두 지원하되, workflow step에서는 기본적으로 stateless를 우선한다.
사용자/프로젝트용 agent 모드에서는 persistent를 허용한다.

#### 12. 완료 기준

다음 조건을 충족하면 1단계 완료로 본다.

워크플로우에서 ClaudeTool 없이도 읽기/검색/부분 수정 계열 작업이 가능하다.
ClaudeTool은 고난도 탐색/코딩 중심으로 역할이 재정의되어 있다.
모든 네이티브 Tool 실행이 감사 가능하다.
write 성격 작업은 정책/승인과 연결된다.
세션이 단순 채팅 로그가 아니라 작업 문맥 저장소로 동작한다.
AXOPM 대표 시나리오 최소 3개가 새 구조로 구현된다.
다른 도메인 앱으로 복제 가능한 onboarding 방식이 정의된다.

#### 13. 주요 리스크와 대응

리스크 1. 범용 플랫폼 욕심

문제:

이것저것 다 하다 보면 OpenWebUI/n8n류와 경쟁하는 모양새가 됨

대응:

범용 기능보다 도메인 앱 재사용성에 집중
"팔릴 제품"이 아니라 "우리 솔루션 공통 AI 운영층" 기준으로 의사결정

리스크 2. ClaudeTool 재구현 과잉

문제:

도구 이식이 아니라 런타임 전체 재구현으로 새어 나감

대응:

1단계는 도구/세션/컨텍스트까지만
고난도 추론/복합 리팩터링은 ClaudeTool 유지

리스크 3. 도구만 만들고 통제가 없음

문제:

### [대화 3] 전체 단계 기획안

좋습니다. 그럼 1단계만 상세화가 아니라, Aimbase 전체 단계 로드맵을 먼저 설계하고, 실제 구현 속도와 순서는 개발팀이 조절할 수 있게 짜드리겠습니다. Aimbase는 이미 워크플로우 DAG, LLM_CALL/TOOL_CALL/HUMAN_INPUT, 세션 저장, MCP 탐색, 정책·감사·사용량, RAG 같은 기반을 갖고 있으므로, 전체 로드맵은 "새 제품 창업"이 아니라 도메인 앱용 AI 운영 레이어로의 재정렬이어야 합니다.

Aimbase 전체 단계 기획안
부제: 도메인 솔루션용 AI 운영 레이어 구축 로드맵

#### 1. 전제

Aimbase의 목적은 범용 AI 툴 시장에서 OpenWebUI, n8n 같은 제품과 경쟁하는 것이 아니다.
Aimbase의 목적은 AXOPM, 입찰 시스템, WMS, OMS, Mall 등 여러 도메인 솔루션에 AI를 빠르고 일관되게 이식하기 위한 공통 운영 레이어가 되는 것이다. AXOPM 문서도 Aimbase를 "앱에 AI를 쉽게 붙이는 AI 엔진"으로 설명하고 있고, MCP 도구와 RAG, 정책, 보안을 통해 업무 액션과 연결하는 구조를 전제로 한다.

따라서 전체 단계의 핵심은 아래 3가지다.

하위 런타임은 교체 가능해야 한다
상위 통제권은 Aimbase가 가져야 한다
도메인 앱에 붙일 때 재사용성이 높아야 한다

#### 2. 전체 전략 한 줄 요약

ClaudeTool/CLI의 장점은 활용하되,
Aimbase는 도구·세션·컨텍스트·정책·감사·워크플로우를 장악하는 상위 운영 레이어가 된다.

#### 3. 전체 단계 개요

전체는 5단계로 설계합니다.

단계 1. Native Tool Foundation

목표:
ClaudeTool 내부 블랙박스에 있던 일부 기능을 Aimbase 네이티브 Tool로 끌어온다.

핵심 산출물:

FileRead / Glob / Grep / SafeEdit(제한형)
Tool Contract
Tool Audit / Approval 연계
ToolRegistry 확장
Tool 노드 UX 강화

의미:
Aimbase가 "ClaudeTool 래퍼"가 아니라 도구 오케스트레이터가 되기 시작하는 단계다.
현재 Aimbase는 이미 TOOL_CALL, 워크플로우 실행, MCP 도구 탐색, 정책/승인 구조를 갖고 있어 이 단계를 바로 시작할 기반이 있다.

단계 2. Session Meta & Context Assembly

목표:
세션을 단순 채팅 로그가 아니라 작업 문맥 저장소로 만들고, 컨텍스트를 Aimbase가 상위에서 조립한다.

핵심 산출물:

project/workspace/session scope
runtime kind / mode 저장
tool execution lineage
session summary 확장
context recipe engine
recent chat + tool result + MCP + RAG + workflow output 조립

의미:
이 단계부터 Aimbase는 "프롬프트 전달기"가 아니라 문맥 운영 엔진이 된다.
문서상 Aimbase는 이미 세션 영구 저장, 히스토리 조회, 컨텍스트 트리밍과 요약 기반 관리 기능을 갖고 있으므로, 이 단계는 새 개념 도입이 아니라 기존 기능의 승격이다.

단계 3. Runtime Orchestration

목표:
ClaudeTool을 포함한 하위 런타임을 상위 통제 안에서 재배치한다.

핵심 산출물:

ClaudeTool mode 분리
stateless
persistent project session
runtime capability profile
runtime adapter interface
workflow 단계별 runtime selection
audit-linked runtime execution

의미:
ClaudeTool은 계속 쓰되, 모든 책임을 떠맡지 않게 만든다.
이 단계가 되면 ClaudeTool은 "모든 걸 대신하는 블랙박스"가 아니라 고난도 자율 작업 전용 런타임으로 정리된다.

단계 4. Domain Pack Standardization

목표:
AXOPM에서 검증한 AI 기능을 다른 솔루션으로 빠르게 복제할 수 있게 만든다.

핵심 산출물:

domain adapter spec
domain context recipe pack
domain workflow template pack
domain tool pack
approval/policy defaults by domain
onboarding bootstrap kit

예시:

AXOPM Pack
입찰 Pack
WMS Pack
OMS Pack
Mall Pack

의미:
이 단계부터 Aimbase는 "특정 프로젝트용 엔진"이 아니라 여러 솔루션에 반복 적용 가능한 AI 운영 기반층이 된다. AXOPM 문서에 이미 프로젝트 건강도, 이슈 triage, 리스크, 업무일지, OPM 6단계 생성 같은 명확한 시나리오가 있으므로, AXOPM이 첫 번째 표준 Pack이 된다.

단계 5. Enterprise Operating Layer Maturity

목표:
운영성과, 거버넌스, 평가, 재현성을 강화하여 내부 표준 플랫폼 수준으로 끌어올린다.

핵심 산출물:

full trace / replay
structured evaluation
tool/run cost attribution
approval simulation
policy simulation
quality benchmark by domain
tenant/app별 운영 대시보드 고도화

의미:
이 단계는 "기능 추가"보다 운영 체계 완성에 가깝다.
문서상 Aimbase는 이미 감사 로그, 액션 로그, 사용량 통계, 평가 API, 플랫폼 모니터링 기능을 보유하거나 예정하고 있으므로, 이 단계는 그 기능들을 제품 철학에 맞게 엮는 단계다.

#### 5. 단계 간 관계

중요한 점은, 이 단계들이 "연 단위 장기계획"이 아니라 계층 순서도라는 점입니다.

즉:

1단계 끝나고 1년 뒤 2단계가 아니라
개발팀은 1→2→3→4→5 순서로 병렬/압축 실행할 수 있습니다

하지만 방향은 반드시 아래에서 위로 갑니다.

Tool 없이 Session만 고도화하면 쓸모가 약함
Session 없이 Context 조립하면 기준이 불명확함
Context 없이 Runtime 추상화하면 ClaudeTool 재구현에 빠짐
Domain Pack 없이 범용화하면 산으로 감
Operating Layer 없이 운영 규모가 커지면 통제 불가

즉, 물리적 시간의 점진이 아니라 구조적 순서의 점진입니다.

2) 세션 메타

이건 문서의 단계 2. Session Meta & Context Assembly 중 앞부분입니다.

여기서 넣은 내용:

session scope type
runtime kind
workspace ref
persistent flag
summary version
context recipe id
tool chain lineage

즉, 우리가 말한 "세션 메타 다음"이 들어가 있습니다.

3) 컨텍스트 조립

이것도 문서의 단계 2 안에 포함되어 있습니다.

여기서 넣은 내용:

session summary
recent turns
tool results
MCP outputs
workflow outputs
RAG results
domain entity context
runtime-specific final formatting
context recipe engine

즉, "도구 결과와 세션 정보를 바탕으로 Aimbase가 상위에서 컨텍스트를 조립한다"는 내용이 들어가 있습니다. Aimbase는 이미 세션 저장, 컨텍스트 트리밍, 워크플로우, RAG, MCP 같은 기반을 갖고 있어서 이 단계가 자연스럽게 이어지도록 썼습니다.

4) 자율 런타임

이건 문서의 단계 3. Runtime Orchestration입니다.

여기서 넣은 내용:

ClaudeTool stateless / persistent mode
runtime adapter interface
runtime capability registry
workflow/runtime routing rules
ClaudeTool의 역할 재정의
ClaudeTool = 고난도 자율 탐색/다중 파일 추론/복합 리팩터링 담당

즉, 우리가 말한 "마지막에 자율 런타임 일부 흡수 또는 재배치"가 들어가 있습니다.

결론

그래서 전체 흐름은 분명히 들어가 있습니다.

도구 → 세션 메타 → 컨텍스트 조립 → 자율 런타임

이 순서가 문서 안에서는 이렇게 변환되어 있습니다.

1. Native Tool Foundation
2. Session Meta & Context Assembly
3. Runtime Orchestration

그리고 그 위에,

4. Domain Pack Standardization
5. Enterprise Operating Layer Maturity

를 더 얹은 구조입니다.

즉, 질문하신 핵심 4단계는 문서의 앞 3개 단계 안에 그대로 포함되어 있고, 제가 그 뒤를 확장 단계로 붙여둔 것입니다.

더 정확히 하려면 문서 제목 체계 자체를 아예 이렇게 바꾸는 게 좋습니다.
